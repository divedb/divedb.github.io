    LevelDB复现之版本控制

    发表于 2025-06-06 分类于 db

    🧩 概要

    🔍 struct FileMetaData字段详解：

    🧱 字段详解：

    字段名 类型 含义与作用
    refs int 引用计数器：用于管理内存中的 FileMetaData 对象的生命周期，确保它不会被过早释放。
    allowed_seeks int 允许的 seeks 次数（在触发 compaction 前）： 用于优化触发 compact 的时机。每当一个查询需要进行二分查找（seek）某个文件时，此计数会递减；当次数耗尽时会触发该文件的 compaction。 初始值为 2^30，表示无需紧急 compact。
    number uint64_t SST 文件编号：唯一标识一个 SST 文件，对应磁盘文件名中的数字部分，如 000123.ldb。
    file_size uint64_t SST 文件大小（以字节为单位）：用于统计和判断是否需要 compact（比如小文件可能被合并）。
    smallest InternalKey SST 文件中最小的 InternalKey： 用于 key range 管理和快速查找（比如确定哪些文件与某个查询 key 有重叠）。
    largest InternalKey SST 文件中最大的 InternalKey： 与 smallest 一起组成该文件的 key 范围。

    在 LevelDB 中，Version 是一个非常核心的类，它表示 某一时刻整个数据库的快照，也就是整个 LSM 树的“视图”。每个 Version 都记录了当前数据库的所有 SST 文件在各个 Level 上的分布信息，支持读操作（Get）、迭代器构建（AddIterators）、重叠检测（GetOverlappingInputs）等功能。

    我们可以把 Version 看作是数据库内部在某一时刻的 元数据状态快照，而 VersionSet 管理着这些快照的演进。

    🧠 TableCache 是什么？

    在 LevelDB 和 RocksDB 中，SST 文件是存储在磁盘上的。
    为了避免频繁打开/关闭文件、重复解析数据结构，它们设计了一个轻量级的缓存系统，即 TableCache。

    🧱 TableCache 的作用总结

    功能 说明
    📁 缓存 SST 文件的 Table 对象 减少打开文件（RandomAccessFile）的频率
    🔢 按 file number 快速定位 所有 SST 文件都有唯一编号，TableCache 是 file_number → Table 的映射
    🔍 加快读取速度 一次打开 SST 后，解析出的 index、filter block 会缓存在内存中
    💥 节省内存使用 LRU 缓存机制，避免所有文件都常驻内存

    🧬 TableCache 在读取流程中的位置

    假设你执行一次 Get(key)，底层流程如下：

    1. MemTable 查找（可能命中）
    2. Immutable MemTable 查找（可能命中）
    3. SST 查找：
      • 根据 Version 里的 files_[] 决定要读哪些文件
      • 每个文件用 file_number 查找 TableCache
      • 如果命中：直接用已有的 Table* 访问
      • 如果没命中：
        • 打开文件 → RandomAccessFile
        • 解析 footer → block handle → index/filter block
        • 创建 Table 对象
        • 加入 TableCache

    ✅ 所以，TableCache 是 SST 文件到解析后的 Table 实例之间的缓存层，避免每次都从头加载。

    📌 举例说明

    假如你有 1000 个 SST 文件，查询一个 key foo：

    • 不使用 TableCache：每次都要打开文件、解析 footer、index block 等，非常慢。
    • 使用 TableCache：
      • 查询时命中缓存：O(1) 取出 Table*，直接查找
      • 不命中：只做一次解析，然后缓存起来，后续复用

    🧼 补充一点：和 BlockCache 区别？

    名称 缓存对象 默认大小 作用
    TableCache SST 文件的 Table 实例 通常为打开文件数量 缓存文件和结构
    BlockCache 数据块、filter block 可配置为几百 MB 缓存实际的 key-value 数据块

    ✅ 总结

    TableCache 是 LevelDB 和 RocksDB 中的 SST 文件缓存层，负责将 file_number 映射到已经打开并解析好的 Table 实例，避免重复 I/O 和结构构建，是提升读取性能的关键组件。
