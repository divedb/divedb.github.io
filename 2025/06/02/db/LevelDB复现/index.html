    写入逻辑

    ​ 在LevelDB中，writers_是一个写请求队列，多个线程可能同时调用DBImpl::Write(...)方法，但只有最前面的那个 Writer（writers_.front()）有资格执行写入操作。其他线程只能等待前面的写入完成。

    ✅ 那是谁调用 writers_ 中真正的写入逻辑呢？

    👉 是队列头部的 Writer对应的线程 —— 也就是当前调用 Write(...) 的线程自己在处理！

    LevelDB 没有专门的写线程或后台线程来做这件事。

    具体来说：

    1. 多个线程同时调用 Write(...)，都往 writers_ 加入自己的 Writer。

    2. 每个线程加完以后判断自己是不是头部：

    3. 只有最前面的那个线程会继续执行主写入逻辑，包括：

      • 构建批次（BuildBatchGroup()）；
      • 写入 WAL（log_->AddRecord(...)）；
      • 同步磁盘（logfile_->Sync()）；
      • 写入 MemTable；
      • 更新序列号。

    4. 写完后，它会负责把后续所有批次的 Writer.status 和 Writer.done 设置好，并通过 cv.Signal() 依次唤醒后面的写线程。

    🔁 举个例子

    假设有三个线程同时写入：

    线程 操作 是否执行写入
    T1 调用 Write()，加入队列 ✅ 是第一个，执行写入
    T2 调用 Write()，加入队列 ⏸ 等待唤醒
    T3 调用 Write()，加入队列 ⏸ 等待唤醒
    • T1 执行完写入，会把 T2 和 T3 的 done = true; status = T1的写入状态; 并唤醒它们。
    • T2 和 T3 醒来发现 done = true，返回状态，整个流程结束。

    💡 调用链简述：

    1. DBImpl::MaybeScheduleCompaction()
      判断是否需要调度 Compaction（例如 L0 文件数量超过阈值）。
    2. DBImpl::BackgroundCall()
      是后台线程调用的入口，决定调用哪个具体任务。
    3. DBImpl::BackgroundCompaction()
      创建 Compaction 对象（通过 VersionSet::PickCompaction()），并调用 DoCompactionWork() 来实际执行。
    4. DBImpl::DoCompactionWork(Compaction\* c)
      将多文件合并为新的 SST，并写入磁盘，同时更新元信息。
