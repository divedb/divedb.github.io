    📘 btrfs设计

    ​ btrfs的实现依赖于简单且广为人知的结构。它在性能方面表现良好，但其长期目标是在文件系统不断增长和老化的过程中保持性能稳定，这比短期跑分表现更为重要。为此，btrfs使用基准测试来模拟整个文件系统生命周期中的性能表现。

    📗 B树数据结构

    ​ btrfs的B树是一个通用的数据存储机制，能够存储多种不同类型的数据。在内部，它仅识别三种结构：

    • 键（key）
    • 项（item）
    • 块头（block header）

    树的上层节点只包含[key, 块指针]对。而树的叶子节点被划分为两个相对生长的区域：一侧是固定大小的item项数组，另一侧是可变大小的item数据区。每个item的offset和size字段指定了该项数据在叶子节点中的存储位置。

    每个item的数据大小不固定，btrfs使用不同类型的item数据来定义各种文件系统结构。btrfs_disk_key中的type字段指明了该项存储的数据类型。

    块头结构中包含了如下内容：

    • 当前块的内容校验和
    • 所属文件系统的UUID
    • 当前块在树中的层级
    • 块号（即该块应被存储的位置）

    这些字段可在读取数据时对元数据进行完整性验证。所有指向btree块的引用也会一并记录其预期的generation字段，这有助于btrfs检测出媒介上的幽灵写入或误写入的情况。

    为了简化文件系统的回写代码，上层节点指针中不会存储下层节点的校验和。节点的generation字段在其被插入B树时就已确定，而校验和只有在实际写盘前才会计算。因此，只用generation字段就可以让btrfs检测到幽灵写入，而无需在每次更新校验和时都回溯并修改上层节点。

    generation字段对应的是分配该块时的事务ID，这不仅支持增量备份，也被写时复制（copy-on-write）事务子系统所使用。

    📚 文件系统数据结构

    ​ 文件系统中的每个对象都有一个objectid，在创建时动态分配。一个尚未使用的objectid表现为文件系统B-tree键空间中的一个“空洞”——即该objectid在树中尚不存在。objectid占据键中最高有效位，使得属于同一个对象的所有项目都可以在B-tree中逻辑上聚集在一起。

    键中的offset字段表示对象中某项数据的字节偏移量。对于文件数据块（extents），该偏移量表示该数据块在文件中的起始位置。type字段表示该项的类型信息，并预留了扩展空间以备将来使用。

    🍃 inode节点

    ​ inode节点存储在struct btrfs_inode_item中，其键的 offset 固定为 0，type 类型值为 1。对于某个对象，inode 项在 B-tree 中始终具有最低的键值。它存储传统的 stat 数据（如文件大小、权限、时间戳等），但不包含嵌入式文件内容或扩展属性数据，这些数据保存在其他类型的项中。

    文件

    小文件（尺寸小于一个 B-tree 叶子节点块）可以直接打包存储在 extent 项中。此时，键的 offset 表示文件中该数据的字节偏移，struct btrfs_item 的 size 字段记录实际存储的数据大小。一个文件可以包含多个这样的项目。

    大文件的数据则以 extent 形式存储。struct btrfs_file_extent_item 记录该数据块的版本号（generation）及其对应的磁盘位置（磁盘块起始地址及块数）。Extent 项还记录了该数据块在文件中的逻辑偏移及其使用的磁盘块数量。这种方式允许 Btrfs 在无需读取旧数据的情况下直接覆盖某段数据。例如，在一个 128MB 的 extent 中间写入 1MB 数据，可能会生成如下三个 extent 项：

    • [旧 extent: 0-64MB]
    • [新 extent: 1MB]
    • [旧 extent: 65MB–128MB]

    文件数据的校验和（checksum）存储在专用的 B-tree 中，结构为 struct btrfs_csum_item。校验和键的 offset 对应于 extent 的起始字节编号。校验在数据压缩或加密之后进行，反映实际写入磁盘的字节。

    一个校验项可能包含多个校验和。注意，内嵌于 B-tree 中的小文件数据由所在 B-tree block 的校验和覆盖。

    引用计数的 Extent

    引用计数是快照子系统的基础。每个分配给 B-tree 或文件的数据块（extent）都会在 struct btrfs_extent_item 中记录引用数量。保存这些项的树结构也充当了磁盘使用情况的分配图。

    部分树（如 extent tree 和树根目录）不会对 extent 进行引用计数，仅依赖写时复制日志保护。不过，磁盘上所有分配块都使用相同类型的 extent item。

    Extent 块组

    Extent block group 将磁盘划分为至少 256MB 的块区，便于优化分配策略。每个 block group 记录可用块的数量。文件和目录优先尝试其所属的首选块组。

    Block group 带有标志位，表示其适用于数据或元数据分配。在格式化时，磁盘会被划分为交错的元数据组（约占 33%）和数据组（约占 66%）。尽量避免数据和元数据混存，提高 fsck 检查效率并减少挂载期间写回的寻道开销（但会略微增加读取时的寻道次数）。

    Extent 树与设备映射（DM）集成

    Btrfs 的 extent 树用于灵活划分磁盘空间，可为某个或一组树根、目录或 inode 提供特定分配策略。管理员可控制文件系统的哪些部分使用条带、镜像或绑定到指定设备。

    Btrfs 计划通过与设备映射器（DM）集成，简化大规模存储池管理。原则上每个磁盘可拥有一个 extent 树，管理员可将其分配给子卷、目录或文件。

    显式反向引用（Back References）

    反向引用有三个主要目的：

    1. 精确追踪引用者：防止错误释放仍被引用的 extent。
    2. 定位引用者：当发现某个块损坏时，能快速找出使用它的对象。
    3. 支持迁移与维护：例如文件系统收缩或存储池维护时迁移块。

    文件 extent 的反向引用

    一个文件 extent 可能被以下引用：

    • 多个快照、子卷，或同一子卷的不同版本
    • 同一子卷中不同文件
    • 同一文件的不同偏移

    旧的 extent_ref_v0 结构（现已不再使用）包含以下字段：

    • 子卷根的 objectid
    • 引用树的 generation
    • 引用该 extent 的文件的 objectid
    • 文件中引用的 offset

    B-tree extent 可被：

    反向引用包含四个 64 位字段，会被哈希成一个 64 位值作为 key 的 offset。key 的 objectid 为 extent 的起始字节，key 类型为 BTRFS_EXTENT_REF_KEY。

    若哈希冲突（offset 溢出），会通过增加 hash 值继续向前查找，直到匹配项出现。

    快照与子卷

    子卷（subvolume）本质上是一个命名的 B-tree，保存文件与目录。它们有独立 inode，可设置属主与属组。子卷可以设置写入配额，一旦超限禁止写入。子卷中所有块与 extent 都使用引用计数以支持快照功能。最多可创建 2⁶⁴ 个子卷。

    快照与子卷结构完全相同，只是在创建时其根块（root block）初始共享另一子卷的根。创建快照时，会增加原根块的引用计数。写时复制机制确保原子性修改，互不干扰。

    📝 btrfs_super_block结构

    Linux版本v4.4.1，btrfs_super_block结构定义在fs/btrfs/ctree.h:453。

    📦 基础头部（前 4 项和btrfs_header匹配）

    字段名 类型 说明
    csum[BTRFS_CSUM_SIZE] 校验和 对整个super block内容的校验值
    fsid[BTRFS_FSID_SIZE] UUID 文件系统的唯一标识符
    bytenr 起始地址 当前super block所在磁盘偏移，一般是64KiB（0x10000）等位置
    flags 标志位 保留字段，目前未使用

    📚 树根指针（Tree Roots）

    字段名 类型 说明
    magic 魔术值 固定为"BTRFS\0" 的十六进制值，用于识别super block
    generation 超级块代数 super block所属的事务ID（可理解为版本号）
    root 根树地址 指向root tree的地址（保存inode tree和目录结构）
    chunk_root Chunk树地址 指向chunk tree，用于管理physical<=>logical block映射
    log_root 日志树地址 指向log tree，用于journaling写前日志

    📌 日志 & 空间信息

    字段名 类型 说明
    log_root_transid 日志事务ID 与log_root配合使用，表示最新日志的事务ID
    total_bytes 总字节数 文件系统总容量（所有设备和）
    bytes_used 已使用字节数 用户数据 + 元数据使用量
    root_dir_objectid 根目录objectid 通常是BTRFS_FIRST_FREE_OBJECTID = 256
    num_devices 设备数量 文件系统中包含的物理设备数量

    📐 参数设定（block大小等）

    字段名 类型 说明
    sectorsize 扇区大小 一般为4K，逻辑IO单位
    nodesize node大小 B-tree节点大小（常见为16K）
    __unused_leafsize 已废弃 早期用于leaf size，现在未使用
    stripesize RAID条带大小 多设备下RAID条带宽度
    sys_chunk_array_size 系统chunk数组长度 用于引导chunk tree所需的mapping信息大小

    📍 Chunk树额外信息

    字段名 类型 说明
    chunk_root_generation chunk树事务ID chunk_root所属的事务ID
    compat_flags 向后兼容特性标志 旧内核可以安全忽略的新功能标记
    compat_ro_flags 只读兼容特性标志 新功能必须挂载为只读
    incompat_flags 不兼容标志 必须支持的新功能，否则无法挂载（如skinny-metadata）
    csum_type 校验算法 当前只支持CRC32C（值为1）
    root_level root tree层级 B-tree层级（如0表示只有leaf）
    chunk_root_level chunk tree层级 同上
    log_root_level log tree层级 同上

    字段名 类型 说明
    btrfs_dev_item dev_item 设备信息 当前设备的信息（如device ID、UUID、size等）
    label[BTRFS_LABEL_SIZE] 卷标 可读写的标签名（例如通过btrfs filesystem label 设置）

    字段名 类型 说明
    cache_generation 缓存事务ID metadata block的cache generation ID，用于加速挂载

    字段名 类型 说明

    🧱 1. 超级块（Superblock）

    超级块是 Btrfs 文件系统的入口，包含关键的元信息。Btrfs 在多个位置（bytenr 为 0x10000、0x4000000、0x400000000）都备份了超级块。

    超级块中包括：

    • magic: _BHRfS_M（标识 Btrfs）
    • generation: 文件系统更新的次数（例如事务编号）
    • root: 指向 root tree 的地址（bytenr）
    • chunk root, device root, extent root 的地址
    • UUID：文件系统 UUID
    • dev_item：本设备信息
    • csum type：校验和算法等

    超级块是 bootstrapping 信息的核心，它告诉我们从哪里开始读取整个元数据结构。

    🌳 2. Root Tree（根树）

    超级块中的 root 指向的是 Root Tree，这是一个包含所有“顶层 B-tree”的索引表：

    每个 item 都是一个 (key, root_item) 对，记录了子 B-tree 的位置。例如：

    key 对应的 tree
    (EXTENT_TREE, ROOT_ITEM, 0) extent tree
    (CHUNK_TREE, ROOT_ITEM, 0) chunk tree
    (DEV_TREE, ROOT_ITEM, 0) device tree
    (CSUM_TREE, ROOT_ITEM, 0) checksum tree
    (FS_TREE, ROOT_ITEM, 0) 文件系统子卷
    (UUID_TREE, ROOT_ITEM, 0) uuid 映射树

    📦 3. 各类子树（子 B-tree）

    这些都是真正记录文件系统细节的结构：

    3.1 Extent Tree

    • 管理物理空间（extent）的分配情况
    • 哪些 block 是 free/allocated

    3.2 Chunk Tree

    • 管理 logical address ↔ physical address 映射
    • 支持 RAID 等功能
    • 把逻辑地址映射到 device/offset

    3.3 Device Tree

    • 管理多设备（multi-device）的元数据
    • 每个设备的大小、UUID、device id 等

    3.4 Checksum Tree

    • 记录每个 block 的校验和（如果启用）

    3.5 File System Tree（FS_TREE）

    • 每个 subvolume 都是一个 FS_TREE
    • 包含 inode、dir、dir_item、extent_data 等

    3.6 UUID Tree

    • 把 subvol 的 ID 映射到 UUID
