    一个简单文件系统实现

    发表于 2025-06-05 分类于 linux

    🧩 概要

    ​ 本文介绍了一个非常简单但功能完整的Linux文件系统，名为uxfs。这个文件系统远未完善，但它提供了足够的接口和功能，使用户能够创建目录、常规文件，以及对常规文件进行读写操作。它还提供了一个mkfs命令和一个简单的fsdb命令。该文件系统存在一些已知缺陷，章节末尾还提供了一些练习，鼓励读者动手实践、修复这些缺陷，并扩展其功能。

    本章为读者提供了探索真实文件系统所需的工具，包括如何下载并编译Linux内核源码，如何编译并加载文件系统模块的说明。此外，还详细讲解了如何使用printk()、kdb和gdb等工具来调试和分析文件系统在内核中的运行流程。uxfs的布局非常小巧，新文件系统甚至可以直接创建在软盘上，方便缺乏Linux使用经验的读者在不需要分区或重新分区磁盘的前提下进行实验。

    本章后面给出完整uxfs文件系统的源代码，该代码已在标准的Linux 2.4.18内核上成功编译并运行。然而，Linux内核更新频繁，新版本的发布很快会让当前版本过时。为避免这种问题，可以访问以下网站：

    www.wiley.com/compbooks/pate

    该网站提供了适用于较新Linux内核版本的uxfs文件系统源代码，同时还附带了在常见Linux发行版上构建uxfs文件系统的详细说明。读者可以自行下载和编译内核源码，并加载该文件系统进行实验。按照此方法，从下载源代码到编译并加载模块，整个过程一般不超过5到10分钟。

    设计新文件系统的思路

    ​ 设计uxfs文件系统的目标是追求极致的简洁性。哪怕是研究一些体积较小的Linux文件系统，初学者仍然需要花费大量时间理解其工作原理。相比之下，uxfs的核心理念就是“小”，仅保留最基本的必要功能。它支持层级式命名空间，并具备创建文件、读写文件等基本操作。为了简化代码，并为读者提供练习空间，一些功能（如重命名、符号链接创建）被有意省略。

    任何对该文件系统进行细致研究的读者都会注意到，它虽然功能完整，但存在许多“空白点”。文件系统的整体结构如图 14.1 所示，其主要设计要点如下：

    • 文件系统使用固定大小的512字节块，由头文件ux_fs.h中的常量UX_BSIZE定义。

    • 文件系统块数量是固定的，除去超级块和inode区块，数据块总数为470个，由UX_MAXBLOCKS定义。

    • inode数量固定为32个（常量 UX_MAXFILES）。其中inode 0和1保留，inode 2分配给根目录，inode 3分配给lost+found目录，剩下的28个inode可供用户使用。

    • 超级块存储在块号0的位置，占用一个块。超级块中包含两个数组，分别用于记录inode和数据块的使用状态，从而避免了位图操作，使源代码更易于理解。超级块还维护了当前空闲inode和空闲数据块的数量。

    • 每个数据块对应一个inode。第一个inode存储在第 8 块。由于inode 0和1被保留，根目录的inode存储在第10块，lost+found目录的inode存储在第11块。剩余的inodes分布在第12到第39块之间。

    • 第一个数据块从第33块开始。当文件系统被创建时，第50块用于存储根目录的目录项，第51 块用于存储lost+found目录的目录项。

    • 每个inode最多只包含9个直接数据块指针，因此最大文件大小被限制为9 × 512字节，即4608字节。

    • 目录项大小固定，包含一个inode编号和一个28字节的文件名。每个目录项的总大小为32字节。

    设计文件系统，需要确定要支持哪些内核接口。除了读取与写入常规文件、创建与删除目录之外，还需决定是否支持硬链接、符号链接、重命名等功能。

    为此，需要了解文件系统可以导出的各种操作函数集合。主要有以下四种操作结构需要导出：

    • super_operations
    • file_operations
    • address_space_operations
    • inode_operations

    除了决定哪些操作由uxfs自己实现外，还可以选择使用内核提供的通用（generic）函数来代替自定义实现，这在很大程度上简化了文件系统的开发工作。

    获取 Linux 内核源码

    ​ 本节介绍如何下载Linux内核源码，如何在源码树中定位与文件系统开发相关的关键文件。后续章节将介绍如何根据你的硬件配置内核、编译它并安装新内核。关于引导程序的配置，包括LILO和GRUB，也会有所涉及。

    你可以从以下网站获取Linux内核源码：

    www.kernel.org

    该网站主页会显示最新的内核版本。例如，在本书撰写时，页面上显示的最新稳定版本如下：

    The latest stable version of the Linux kernel is: 2.4.18 2002-07-10 00:40

    UTC F V VI Changelog

    网站还会标注不同内核版本的状态，包括哪个是当前稳定版。点击内核版本链接即可下载最新版本内核；点击changelog可查看该版本的更新内容。

    ​ 该网站收录了从Linux初期至今的所有内核版本。你可以通过导航链接访问源码仓库，选择所需的内核版本。如果你希望使用书中提供的源代码，则需下载2.4.18内核。另一方面，如前所述，也可以从以下网站获取与更新版本兼容的uxfs文件系统代码：

    www.wiley.com/compbooks/pate

    该网站还提供了关于uxfs所支持的Linux内核版本和发行版的信息。

    如下地址：

    www.kernel.org/pub/linux/kernel/v2.4/

    内核源码以.tar.gz格式压缩打包。下载完成后，需先解压再解包。虽然并非强制，但推荐将源码解包至/usr/src目录。可执行如下操作：

    1. 将下载的压缩文件放入/usr/src；
    2. 进入该目录后解压并解包；
    3. 可选择建立一个符号链接指向解包后的目录，方便编译和访问。

    默认情况下，从tar包中解压出的文件会被放在当前工作目录下的linux目录中。将旧的linux目录移走这一命令可以根据开发环境进行调整。另外，也可以选择在一个单独的目录中解压源代码，然后再将其移动到/usr/src/linux-2.4.18目录中。注意，不要覆盖已有的Linux内核源代码树。

    Linux内核源码树

    Linux内核源码树中包含了大量的文件与目录。本节简要介绍源码树的结构，帮助读者快速定位各种内核子系统及特定文件。

    • arch
      包含针对不同硬件架构（如Intel、Sparc、MIPS和IBM s390等）的子目录。

    • CREDITS
      该文件列出了对内核开发做出重大贡献的人员，以及他们的专长领域或贡献内容。

    • Documentation
      包含丰富的内核文档。其中的filesystems目录介绍了多个Linux文件系统及一些通用的文件系统相关知识。

    • drivers
      存放所有Linux设备驱动程序的目录。

    • fs
      与文件系统开发者最相关的目录之一。这里包含了如dcache（目录项缓存）、buffer cache（缓冲区缓存）、inode cache（索引节点缓存）等实现代码，以及与文件相关的系统调用处理。每个文件系统对应一个子目录，源码文件就放在各自目录下。

    • include
      存放所有内核头文件。该目录下既有与架构相关的头文件，也有适用于所有架构的通用头文件，后者主要位于 include/linux/子目录中。对文件系统开发者而言，fs.h头文件尤为重要；而dcache.h定义了Linux dcache所使用的结构体。

    • init
      包含内核启动时执行的初始化函数。

    • ipc
      包含System V IPC（进程间通信）相关的源代码，包括信号量、共享内存和消息队列等功能。

    • kdb
      如果应用了kdb补丁，该目录将包含内核调试器（kernel debugger）的源代码。需要注意的是，kdb补丁还会修改内核中的其他多个文件。

    • kernel
      包含核心内核功能的实现，如进程管理、系统调用处理、模块管理等。

    • lib
      一些标准C库中的函数在内核中也有对应的实现，其源码位于此目录。

    • MAINTAINERS
      该文件列出了对内核不同部分负责的维护人员及其联系信息。

    • mm
      包含与架构无关的内存管理代码。Linux的页缓存（page cache）管理相关代码也在该目录中。

    • net
      包含所有网络协议（如TCP、UDP、IP等）的实现。

    由于内核源码中的文件与目录数量庞大，这里无法一一列举。不过，对于有兴趣研究文件系统的读者来说，include/、fs/和mm/是最核心的几个目录，绝大多数与文件系统相关的结构体与函数都在其中。此外，如果你想深入了解文件系统与驱动程序之间的接口，也可以查阅drivers/block/目录中的一些关键文件。

    补充：

    1. 用户命名空间（user namespace）

      ​ User Namespace是Linux内核的一项功能，它允许进程拥有与其在宿主机上不同的用户ID (UID) 和组ID (GID) 映射。这意味着一个进程可以在其自己的User Namespace内拥有root权限（UID 0），但在宿主机上却是一个普通用户，从而实现权限的隔离。

      User Namespace是在Linux Kernel 3.8版本中引入的。从Linux 3.8开始，非特权进程也可以创建User Namespace，这为容器技术的发展提供了重要的基础。

      解决了什么问题？

      1. 权限隔离和安全性增强：
        • 在传统的Linux系统中，如果一个进程需要执行某些特权操作（例如，挂载文件系统），它通常需要root权限。然而，root权限是全局的，如果一个拥有 root 权限的进程被攻破，攻击者就可以控制整个宿主机系统。
        • User Namespace允许进程在命名空间内部被认为是root用户，但实际上在宿主机上映射到一个非特权用户。这样，即使容器内的root用户被攻破，其权限也仅限于该User Namespace内部，无法对宿主机系统造成危害，大大增强了安全性。
        • 这对于构建“无根容器”（rootless containers）至关重要，它允许普通用户运行容器，而无需root权限或依赖setuid程序，降低了安全风险。
      2. 多用户环境下的灵活权限管理：
        • 在容器化环境中，你可能希望容器内的应用程序以特定的用户身份运行，而不是宿主机上的真实用户。User Namespace允许你灵活地将容器内部的UID/GID映射到宿主机上的不同UID/GID范围，从而实现更精细的权限控制。
      3. 简化特权操作：
        • 一些应用程序在容器内需要以root身份运行才能正常工作（例如，安装软件包）。User Namespace使得这些应用程序可以在隔离的环境中获得必要的特权，而不会影响宿主机的安全性。
