深入理解promise类型

发表于 2025-04-14 分类于 c++ ， co_routine

翻译：C++ Coroutines: Understanding the promise type

这篇文章是关于C++协程TS (N4736)系列的第三篇。

本系列的前几篇文章涵盖了：

  • 协程理论
  • 理解operator co_await

在这篇文章中，我将探讨编译器如何将您编写的协程代码转换为已编译的代码，以及如何通过定义您自己的Promise类型来自定义协程的行为。

协程概念

协程TS添加了三个新的关键字：co_await、co_yield和co_return。在函数体中使用这些协程关键字之一，都会触发编译器将该函数编译为协程，而不是普通函数。

编译器对您编写的代码转换成一个状态机，该状态机允许在函数内的特定点挂起执行，然后在以后恢复执行。

在上一篇文章中，我描述了协程TS引入的两个新接口中的第一个：Awaitable接口。TS引入的第二个对于代码转换很重要的接口是 Promise 接口。

Promise接口用于自定义协程自身行为。库编写者可以自定义在调用协程时发生什么，在协程返回时（无论是正常返回还是通过未处理的异常）发生什么，以及自定义协程中任何co_await或co_yield表达式的行为。

Promise对象

Promise对象通过实现在协程执行期间的特定点调用的方法来定义和控制协程自身的行为。

在我们继续之前，我希望您尝试摆脱对“promise”的任何先入为主的观念。虽然在某些用例中，协程的promise对象确实扮演着与std::promise部分类似的角色，但在其他用例中，这种类比有些牵强。将协程的promise对象视为一个“协程状态控制器”对象可能更容易理解，该对象控制协程的行为并可用于跟踪其状态。

对于每次调用协程函数，都会在协程帧内构造一个promise对象的实例。

编译器会在协程执行期间的关键点生成对promise对象上某些方法的调用。

当调用协程函数时，在执行协程体源代码中的代码之前会执行一些与常规函数略有不同的步骤。

以下是步骤的摘要（我将在下面详细介绍每个步骤）。

  1. 使用operator new分配协程帧（可选）。
  2. 将任何函数参数复制到协程帧。
  3. 调用类型为P的promise对象的构造函数。
  4. 调用promise.get_return_object()方法以获取协程首次挂起时返回给调用者的结果。将结果保存为局部变量。
  5. 调用promise.initial_suspend()方法并co_await其结果。
  6. 当co_await promise.initial_suspend()表达式恢复时（无论是立即还是异步地），协程开始执行协程体语句。

当执行到达 co_return 语句时，会执行一些额外的步骤：

  1. 调用promise.return_void()或promise.return_value(<expr>)。
  2. 以创建的相反顺序销毁所有具有自动存储期的变量。
  3. 调用promise.final_suspend()并co_await其结果。

相反，如果由于未处理的异常导致执行离开<body-statements>，则：

  1. 捕获异常并在catch块内调用promise.unhandled_exception()。
  2. 调用promise.final_suspend()并co_await其结果。

一旦执行传播到协程体之外，协程帧就会被销毁。销毁协程帧涉及以下几个步骤：

  1. 调用promise对象的析构函数。
  2. 调用函数参数副本的析构函数。
  3. 调用operator delete以释放协程帧使用的内存（可选）。
  4. 将执行权返回给调用者/恢复者。

当执行首次到达co_await表达式内的<return-to-caller-or-resumer>点，或者如果协程运行完成而没有到达 <return-to-caller-or-resumer>点时，协程将被挂起或销毁，并且先前从promise.get_return_object()调用返回的返回对象将被返回给协程的调用者。

分配协程帧

首先，编译器会生成对operator new的调用，以分配协程帧的内存。

如果promise类型P自定义了operator new方法，则调用该方法，否则调用全局的operator new。

  • 传递给operator new的大小不是sizeof(P)，而是整个协程帧的大小，并且由编译器根据参数的数量和大小、promise 对象的大小、局部变量的数量和大小以及管理协程状态所需的其他编译器特定的存储自动确定。

  • 如果满足以下条件，编译器可以自由地省略对

    • 它能够确定协程帧的生命周期严格嵌套在调用者的生命周期内；并且
    • 编译器可以在调用点看到所需的协程帧大小。 在这种情况下，编译器可以在调用者的激活帧（堆栈帧或协程帧部分）中为协程帧分配存储空间。

  • 协程 TS 尚未指定任何保证省略分配的情况，因此您仍然需要编写代码，就好像协程帧的分配可能因 std::bad_alloc 而失败一样。这也意味着您通常不应该将协程函数声明为 noexcept，除非您接受在协程无法为协程帧分配内存时调用 std::terminate()。

将参数复制到协程帧

协程需要将原始调用者传递给协程函数的任何参数复制到协程帧中，以便在协程挂起后它们仍然有效。

如果参数通过值传递给协程，则这些参数通过调用类型的移动构造函数复制到协程帧中。

如果参数通过引用（左值或右值）传递给协程，则只有引用被复制到协程帧中，而不是它们指向的值。

请注意，对于具有平凡析构函数的类型，如果参数在协程中可到达的 <return-to-caller-or-resumer> 点之后从未被引用，则编译器可以自由地省略参数的复制。

当通过引用将参数传递到协程中时，涉及许多陷阱，因为您不能总是依赖引用在协程的整个生命周期内保持有效。许多在普通函数中使用的常见技术，例如完美转发和通用引用，如果与协程一起使用，可能会导致未定义的行为。如果您想了解更多详细信息，Toby Allsopp 撰写了一篇很棒的文章。

如果任何参数的复制/移动构造函数抛出异常，则任何已构造的参数都会被销毁，协程帧被释放，并且异常会传播回调用者。

构造 Promise 对象

一旦所有参数都被复制到协程帧中，协程就会构造 promise 对象。

在构造 promise 对象之前复制参数的原因是允许 promise 对象在其构造函数中访问复制后的参数。

首先，编译器检查是否存在可以接受对每个复制参数的左值引用的 promise 构造函数的重载。如果编译器找到这样的重载，则编译器会生成对该构造函数重载的调用。如果找不到这样的重载，则编译器会回退到生成对 promise 类型的默认构造函数的调用。

请注意，promise 构造函数能够“窥视”参数的能力是协程 TS 中相对较新的更改，在 2018 年杰克逊维尔会议的 N4723 中被采纳。有关提案，请参见 P0914R1。因此，某些旧版本的 Clang 或 MSVC 可能不支持它。

如果 promise 构造函数抛出异常，则在异常传播到调用者之前，参数副本将被销毁，并且协程帧在堆栈展开期间被释放。

返回对象 是在协程首次挂起或运行完成后执行返回给调用者时，返回给协程函数调用者的值。

初始挂起点

协程在初始化协程帧并获取返回对象后执行的下一件事是执行语句 co_await promise.initial_suspend();。

这允许 promise_type 的作者控制协程是在执行源代码中出现的协程体之前挂起，还是立即开始执行协程体。

如果协程在初始挂起点挂起，那么以后可以通过在协程的 coroutine_handle 上调用 resume() 或 destroy()，在您选择的时间恢复或销毁它。

co_await promise.initial_suspend() 表达式的结果被丢弃，因此实现通常应该从 awaiter 的 await_resume() 方法返回 void。

重要的是要注意，此语句存在于保护协程其余部分的 try/catch 块之外（如果您忘记了它的样子，请向上滚动到协程体的定义）。这意味着在到达其 <return-to-caller-or-resumer> 点之前，从 co_await promise.initial_suspend() 的评估中抛出的任何异常将在销毁协程帧和返回对象后重新抛给协程的调用者。

如果您的返回对象具有在销毁时销毁协程帧的 RAII 语义，请注意这一点。如果是这种情况，那么您需要确保 co_await promise.initial_suspend() 是 noexcept 的，以避免双重释放协程帧。

请注意，有一个提案正在调整语义，以便 co_await promise.initial_suspend() 表达式的全部或部分位于协程体的 try/catch 块内，因此在协程最终确定之前，这里的确切语义可能会发生变化。

对于许多类型的协程，initial_suspend() 方法要么返回 std::experimental::suspend_always（如果操作是惰性启动的），要么返回 std::experimental::suspend_never（如果操作是迫切启动的），它们都是 noexcept 的 awaitable 对象，因此这通常不是问题。

返回给调用者

当协程函数到达其第一个 <return-to-caller-or-resumer> 点时（或者如果没有到达这样的点，则当协程执行完成时），从 get_return_object() 调用返回的 返回对象 将返回给协程的调用者。

请注意，返回对象 的类型不必与协程函数的返回类型相同。如果需要，将执行从 返回对象 到协程返回类型的隐式转换。

请注意，Clang 的协程实现（截至 5.0）将此转换的执行推迟到从协程调用返回返回对象时，而 MSVC 的实现（截至 2017 Update 3）在调用 get_return_object() 后立即执行转换。尽管协程 TS 对预期行为没有明确说明，但我相信 MSVC 计划更改其实现以更像 Clang 的行为，因为这启用了一些有趣的用例。

使用 co_return 从协程返回

当协程到达 co_return 语句时，它会被转换为对 promise.return_void() 或 promise.return_value(<expr>) 的调用，后跟 goto FinalSuspend;。

转换规则如下：

  • co_return; -> promise.return_void();
  • co_return <expr>; -> <expr>; promise.return_void(); 如果 <expr> 的类型为 void
  • co_return <expr>; -> promise.return_value(<expr>); 如果 <expr> 的类型不是 void

随后的 goto FinalSuspend; 会导致所有具有自动存储期的局部变量以构造的相反顺序被销毁，然后再评估 co_await promise.final_suspend();。

请注意，如果执行在没有 co_return 语句的情况下运行到协程的末尾，则这等效于在函数体的末尾有一个 co_return;。在这种情况下，如果 promise_type 没有 return_void() 方法，则行为是未定义的。

如果 <expr> 的评估或对 promise.return_void() 或 promise.return_value() 的调用抛出异常，则异常仍然会传播到 promise.unhandled_exception()（见下文）。

处理从协程体传播出来的异常

如果异常从协程体传播出来，则该异常会被捕获，并且在 catch 块内调用 promise.unhandled_exception() 方法。

此方法的实现通常调用 std::current_exception() 来捕获异常的副本并将其存储起来，以便以后在不同的上下文中重新抛出。

或者，实现可以通过执行 throw; 语句立即重新抛出异常。例如，请参见 folly::Optional。但是，这样做（很可能 - 见下文）会导致协程帧立即被销毁，并且异常传播到调用者/恢复者。这可能会给一些假设/要求对 coroutine_handle::resume() 的调用是 noexcept 的抽象带来问题，因此您通常只应在完全控制谁/什么调用 resume() 时使用此方法。

请注意，当前的 协程 TS 文档对于如果对 unhandled_exception() 的调用重新抛出异常（或者就此而言，如果 try 块之外的任何逻辑抛出异常）的预期行为有点不清楚。

我对文档的当前理解是，如果控制离开协程体，无论是通过异常从 co_await promise.initial_suspend()、promise.unhandled_exception() 或 co_await promise.final_suspend() 传播出来，还是通过协程运行完成，因为 co_await p.final_suspend() 同步完成，那么协程帧会在执行返回给调用者/恢复者之前自动销毁。然而，这种解释本身也存在问题。

希望未来版本的协程规范能够澄清这种情况。但在那之前，我建议不要从 initial_suspend()、final_suspend() 或 unhandled_exception() 中抛出异常。敬请关注！

最终挂起点

一旦执行离开用户定义的协程体部分，并且结果已通过调用 return_void()、return_value() 或 unhandled_exception() 捕获，并且任何局部变量已被销毁，协程就有机会在执行返回给调用者/恢复者之前执行一些额外的逻辑。

协程执行 co_await promise.final_suspend(); 语句。

这允许协程执行一些逻辑，例如发布结果、发出完成信号或恢复延续。它还允许协程在协程执行完成并且协程帧被销毁之前选择性地立即挂起。

请注意，在 final_suspend 点挂起的协程上调用 resume() 是未定义的行为。您唯一能对在这里挂起的协程做的就是 destroy() 它。

根据 Gor Nishanov 的说法，这种限制的基本原理是，由于减少了协程需要表示的挂起状态的数量以及可能减少了所需的条件分支的数量，这为编译器提供了几个优化机会。

请注意，虽然允许协程在 final_suspend 点不挂起，但建议您尽可能地将协程构建为在 final_suspend 点挂起。这是因为这迫使您从协程外部（通常是从某些 RAII 对象的析构函数中）对协程调用 .destroy()，这使得编译器更容易确定协程帧的生命周期范围是否嵌套在调用者内部。反过来，这使得编译器更有可能省略协程帧的内存分配。

  2. 如果您有对协程的 promise 对象的引用，可以使用 coroutine_handle<Promise>::from_promise() 重建其 coroutine_handle。

在协程在 co_await 表达式的 <suspend-point> 挂起后，等待协程的 coroutine_handle 将被传递给 awaiter 的 await_suspend() 方法。您可以将此 coroutine_handle 视为在 延续传递风格 调用中表示协程的延续。

请注意，coroutine_handle 不是 RAII 对象。您必须手动调用 .destroy() 来销毁协程帧并释放其资源。将其视为用于管理内存的 void* 的等效物。这是出于性能原因：使其成为 RAII 对象会给协程增加额外的开销，例如需要引用计数。

您通常应该尝试使用为协程提供 RAII 语义的更高级别的类型，例如 cppcoro（无耻地宣传一下）提供的类型，或者为您自己的协程类型编写封装协程帧生命周期的更高级别的类型。

自定义 co_await 的行为

promise 类型可以选择性地自定义出现在协程体中的每个 co_await 表达式的行为。

只需在 promise 类型上定义一个名为 await_transform() 的方法，编译器就会将出现在协程体中的每个 co_await <expr> 转换为 co_await promise.await_transform(<expr>)。

这有许多重要且强大的用途：

  1. 它允许您启用通常不可等待的类型的等待。 例如，返回类型为 std::optional<T> 的协程的 promise 类型可以提供一个接受 std::optional<U> 并返回一个 awaitable 类型的 await_transform() 重载，该 awaitable 类型要么返回 U 类型的值，要么在等待的值包含 nullopt 时挂起协程。

    C++

  2. 它允许您通过将 await_transform 重载声明为已删除来禁止等待某些类型。 例如，返回类型为 std::generator<T> 的 promise 类型可以声明一个已删除的 await_transform() 模板成员函数，该函数接受任何类型。这基本上禁用了在协程中使用 co_await。

    C++

  3. 它允许您调整和更改通常可等待的值的行为。 例如，您可以定义一种协程类型，该类型通过将 awaitable 对象包装在 resume_on() 运算符中（请参见 cppcoro::resume_on()），确保协程始终从关联执行器上的每个 co_await 表达式恢复。

    C++

关于 await_transform() 的最后一点是，重要的是要注意，如果 promise 类型定义了任何 await_transform() 成员，那么这会触发编译器将所有 co_await 表达式转换为调用 promise.await_transform()。这意味着，如果您只想为某些类型自定义 co_await 的行为，那么您还需要提供一个 await_transform() 的回退重载，该重载只是转发参数。

自定义 co_yield 的行为

您可以通过 promise 类型自定义的最后一件事是 co_yield 关键字的行为。

如果 co_yield 关键字出现在协程中，那么编译器会将表达式 co_yield <expr> 转换为表达式 co_await promise.yield_value(<expr>)。因此，promise 类型可以通过在其 promise 对象上定义一个或多个 yield_value() 方法来自定义 co_yield 关键字的行为。

请注意，与 await_transform 不同，如果 promise 类型没有定义 yield_value() 方法，则 co_yield 没有默认行为。因此，虽然 promise 类型需要通过声明已删除的 await_transform() 来显式选择不允许 co_await，但 promise 类型需要选择支持 co_yield。

具有 yield_value() 方法的 promise 类型的典型示例是 std::generator<T> 的 promise 类型。
