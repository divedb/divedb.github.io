    库函数vs系统调用

    ​ 系统调用（System calls）是一种将控制权从用户进程转移到操作系统内核的函数。例如，read()和write()就是系统调用。进程传递相应的参数来调用它们，控制权转交给内核，在内核中执行该系统调用，将结果返回给调用进程，最后再将控制权交还给用户进程。

    ​ 库函数（Library functions）通常提供更丰富的功能。例如，库函数fread()可以从文件中读取指定大小的若干数据元素。fread()内部会调用read()从文件中读取数据。

    库函数构建在系统调用之上。使用系统调用还是库函数取决于所编写的应用程序。如果某个应用希望对I/O操作有更细致的控制，以优化性能，它可能需要直接调用系统调用。而如果利用库函数中已经提供的功能，这可以节省大量编程工作。

    系统调用通常比调用库函数消耗更多的时间，因为它们涉及从用户态切换到内核态的过程。然而，库函数的具体实现可能无法满足某个特定应用程序的需求。换句话说，是否使用库函数或系统调用并不是一个显而易见的选择，这取决于所编写的应用程序的具体需求。

    数据与属性缓存（Data and Attribute Caching）

    ​ 在进行文件I/O时，可以通过open()系统调用传入一些标志位（flags）来控制行为。此外，某些文件系统还支持一些额外的、非标准的方法来提升I/O性能。

    依据Single UNIX Specification，有三个标准支持的open()选项会影响随后的I/O操作。写入数据时，系统需要将两个内容写入磁盘：文件数据和inode元数据。Inode是磁盘上存储的一个结构，描述了文件的属性（可通过stat()获取）以及指向该文件所有数据块的块映射。

    以下是三个标准支持的选项：

    • O_SYNC：无论写入是否需要新分配空间，写入操作都会在数据和所有元数据（包括inode）写入磁盘后才返回；对于读取操作，访问时间戳也会在读取返回前同步更新到磁盘。

    • O_DSYNC：写入操作在数据写入磁盘后即可返回，但元数据（如inode）可以延迟写入。这种方式在文件系统实现得当的前提下，能显著减少inode更新次数，因而提升I/O吞吐量。当写入会导致新块分配（如写入“空洞”或超过文件末尾），元数据也会被写入磁盘；但若不涉及分配，通常时间戳等元数据不会同步更新。

    • O_RSYNC：如果设置了O_RSYNC和O_DSYNC，则读取操作会等到读取的数据和文件属性（不包括时间戳）同步更新到磁盘后才返回；如果读取的数据范围与尚未完成的写入重叠，这些写入也会在读取前被提交。

      若设置了O_RSYNC和O_SYNC，行为与O_RSYNC | O_DSYNC相同，但还会要求所有由读操作修改的文件属性（包括所有时间戳）也必须同步到磁盘之后才返回。

    如何选择这些选项取决于具体应用的需求：

    对于I/O密集型应用，如果时间戳的准确性并不关键，那么使用O_DSYNC替代O_SYNC可以带来显著的性能提升。

    其他open选项说明（Miscellaneous Open Options）

    ​ 通过在调用open()函数时传入O_NONBLOCK或O_NDELAY标志，应用程序可以在读写操作可能阻塞的情况下获得额外的控制能力。

    • O_EXCL：如果同时指定了O_CREAT和O_EXCL，当目标文件已存在时，open()调用将会失败。
      如果没有指定O_CREAT，那么单独使用O_EXCL的行为是未定义的。
    • O_NONBLOCK / O_NDELAY：这两个标志会影响后续的读写操作行为。如果同时设置了O_NDELAY和O_NONBLOCK，那O_NONBLOCK优先生效。由于这两个选项主要用于管道（pipe）操作，这里不再展开讨论。
    杂项打开选项（Miscellaneous Open Options）

    通过向open()函数传入O_NONBLOCK和O_NDELAY标志，应用程序在读写发生阻塞的情况下可以获得额外的控制能力。

    • O_EXCL：当O_CREAT与O_EXCL同时设置时，如果目标文件已存在，open()调用将失败。若未设置O_CREAT，传入O_EXCL 的行为是未定义的。
    • O_NONBLOCK / O_NDELAY：这两个标志会影响后续的读写操作行为。如果两者都被设置，则以O_NONBLOCK为准。由于它们主要用于管道（pipes）场景。

    原因：
    在管道（包括匿名管道、FIFO 和某些设备文件）中，默认的读写操作是阻塞的。这意味着：

    • 如果你试图读取一个没有数据的管道，read() 会阻塞直到有数据可读。
    • 如果你试图向一个没有读取端的管道写入数据，write() 也可能阻塞或失败。

    使用 O_NONBLOCK 或 O_NDELAY 可以让程序以非阻塞模式访问这些资源，从而避免卡住主流程。

    文件与记录锁（File and Record Locking）

    ​ 当多个进程同时对同一个文件进行写操作时，其结果是不可预测的。在UNIX内核中，同一时间只能有一个写操作能对同一文件生效。然而，不同进程的执行顺序会受到多种因素影响，因此写入顺序可能会产生差异。显然，这种不可控的行为在实际应用中是非常不理想的，因此有必要在应用层对文件进行加锁，可以是整个文件，也可以是文件的特定区域。文件的部分区域通常被称为“记录（record）”，因此我们称之为“文件与记录锁”。

    文件锁有很多用途。以数据库文件访问为例，这是应用程序所需锁机制的一个经典场景。例如，多个用户应该能够同时读取数据库记录；当某个记录被更新时，其他用户仍应能访问其他记录；最后，记录的更新必须遵循严格的时间顺序。

    锁的类型

    ​ 用于控制文件访问的锁分为两种：强制锁（mandatory lock）和建议锁（advisory lock）。

    • 建议锁（Advisory Lock）允许多个协作进程以受控方式安全地访问同一个文件。
    • 强制锁（Mandatory Lock）的机制比较复杂，有些系统视其为“变通做法”（hack），后面会具体讨论。

    本节主要聚焦于建议锁，又称记录锁（record locking）。

    建议锁（Advisory Locking）

    UNIX系统中有三种常见的函数可以实现建议锁：lockf()、flock()和fcntl()。

    flock()是BSD UNIX中引入的函数，不属于POSIX标准，也不被单一UNIX规范（Single UNIX Specification）支持。它用于对整个文件施加建议锁，函数定义如下：

    • operation参数指定锁的类型：

      • LOCK_EX：排他锁，仅允许一个进程写入；
      • LOCK_UN：解锁。
    • 若希望在锁不可用时不阻塞调用，可以将operation LOCK_NB按位或（OR），这样如果文件已被锁定，函数会返回EAGAIN错误。

    要使用lockf()函数，必须以O_WRONLY或O_RDWR标志打开文件。参数size指定从当前文件指针位置开始需要加锁的字节数。因此，在调用lockf()之前通常需要先调用lseek()设置文件偏移量。如果size的值为0，则表示从当前位置一直加锁到文件末尾。

    lockf()的功能参数可以是以下几种：

    • F_LOCK：设置对文件的独占锁。如果文件已被其他进程加锁，调用进程将会阻塞，直到锁被释放。
    • F_TLOCK：与F_LOCK类似，但不会阻塞。如果文件已被锁定，则立即返回EAGAIN错误。
    • F_ULOCK：解除文件某一段的锁定。
    • F_TEST：测试指定的文件段是否已被加锁。如果没有锁，则返回0；如果存在锁，则返回-1，并将errno设置为 EACCES。

    如果尝试加锁的区域包含了部分或完整的已有锁段，那么最终将形成一个新的、合并的锁段。类似地，如果解锁的区域是原有锁段的一部分，或者覆盖多个锁段，结果可能是一个或多个缩小后的锁段。如果size为 0，解锁范围为当前位置至文件末尾。

    需要注意，若两个进程互相尝试加锁对方已持有的文件段，可能会导致死锁。内核可检测此类情况，并在发生时返回 EDEADLK错误。

    强制锁定（Mandatory Locking）

    ​ 正如前面的例子所示，如果多个进程访问同一文件没有通过建议锁（advisory lock）进行协作，就可能出现不可预测的行为。强制锁定（mandatory locking）是一种设计用来在不协作的进程之间实现文件锁定的机制。

    异步 I/O（Asynchronous I/O）

    ​ 通过异步方式发起I/O操作，应用程序可以在等待I/O完成的同时继续执行其他任务。多年来，异步I/O（通常称为async I/O）已经有了多种不同的实现方式。本节将介绍由Single UNIX Specification所支持的异步I/O接口。

    异步I/O的一个常见应用场景是在Oracle数据库中的写入进程（DBWR）。DBWR是Oracle的核心进程之一，负责管理Oracle缓冲区缓存（buffer cache），即数据库块的用户态缓存。其主要任务包括响应读取请求，以及将“脏页”（即被修改过的数据块）写回磁盘。

    在活跃的数据库系统中，DBWR的任务非常繁重，因为它必须不断地将脏页写入磁盘，从而为新的块读取腾出空间。为了减轻这些性能瓶颈，Oracle采用了两种优化方式：第一种是使用多个DBWR进程（称为DBWR从属进程）；第二种（也是显著提高吞吐量的一种）就是利用异步I/O。如果I/O操作是异步执行的，那么DBWR进程在等待I/O完成期间可以继续执行其他任务，比如刷新更多缓存、读取数据块或进行内部管理等。

    64 位文件访问（LFS：Large File Support）

    ​ 在传统的32位操作系统中，文件偏移量通常使用有符号的long类型表示。这就导致文件的最大大小限制为$2^{31}-1$（即2GB - 1字节）。由于将现有应用程序迁移到使用更大类型（如64位）来表示文件偏移量所需的工作量巨大，开发者长期未能推进这一变更。

    为了解决这个问题，一些操作系统和文件系统的厂商联合发起了Large File Summit（大型文件峰会），旨在制定一份规范，使得在现有系统中能够访问大文件。最终，这项规范被纳入了Single UNIX Specification（从UNIX 95开始）。

    该规范引入了以下几个关键概念：

    • off_t类型将支持多种大小（如32位和64位），以适配操作系统和文件系统向完整的64位解决方案过渡。
    • 提供一种偏移量上限（offset maximum）的机制，作为接口的一部分，用于告知应用程序当前操作系统或文件系统所支持的最大文件偏移量。这个上限可以在open()调用时根据是否启用大文件访问来确定。
    • 如果应用程序尝试读取超出其所理解的最大偏移量范围的文件部分，操作系统将返回新的错误码：EOVERFLOW。

    为了同时支持显式访问大文件和透明、易于迁移的升级方式，规范设计了两种编程模型：

    1. 编译期模型（透明方式）

    通过在编译和链接阶段决定off_t的大小。

    • 这将确定是否使用标准的read()、write()等系统调用，或切换到专门用于大文件的版本（如read64()等）。
    • 无论采用哪种方式，应用程序调用的接口保持不变，由链接器在后台自动进行适配。
    2. 显式模型（手动控制）

    程序员在代码中显式指定使用的偏移类型：

    • 在32位系统中，默认的off_t是32位，无法表示大于2GB的文件。
    • 要支持大文件，程序需要使用off64_t类型，并调用如open64()、read64()等函数，以便能够访问大于2GB的文件。

    稀疏文件（Sparse Files）

    ​ 由于稀疏文件的使用较为罕见，许多人对其理解并不深入，常常引起困惑。例如，VxFS文件系统在3.5版本之前，支持的最大文件系统容量为1TB，但单个文件的最大大小却可以达到2TB。这似乎令人费解：一个文件怎么可能比其所在的文件系统还要大？

    所谓稀疏文件，其本质上是包含一个或多个“空洞（hole）”的文件。这正是许多困惑的根源。“空洞”是指文件中未实际分配数据块的一段区域。比如，一个文件可能由一个1KB的数据块、一个1KB的空洞、再加上一个1KB的数据块构成。尽管文件的逻辑大小为3KB，但实际只分配了两个数据块。读取空洞部分时，系统会返回全零。

    ✅ 总结

    ​ 本章介绍了基于文件I/O的系统调用。理解这些基础概念对于深入掌握文件系统的实现机制至关重要。只有明确用户对文件操作的预期，才能更好地理解内核和各个文件系统的设计目标及其实现方式。

    在UNIX系统上进行编程时，遵循相关标准是一种良好的实践，这有助于程序在不同版本的UNIX系统之间具备良好的可移植性。虽然商业版UNIX通常支持Single UNIX Specification（单一UNIX规范）标准，但该标准在Linux和BSD中尚未被完全采纳。不过，几乎所有UNIX系统版本都至少支持POSIX.1标准。

    补充

    sigevent 和常见的信号处理函数（如 signal()、sigaction()）虽然都与 信号（signal）机制相关，但用途和概念有显著区别：

    🔹 sigevent 是“通知机制的结构体”

    它是一个结构体，常用于 异步 I/O（aio） 和 定时器（timerfd, POSIX timers），指定当事件发生（如 I/O 完成或定时器超时）时，内核通过哪种方式通知你。

    常见的 sigev_notify 值：

    • SIGEV_NONE：不通知
    • SIGEV_SIGNAL：通过 signal 通知（结合 sigev_signo）
    • SIGEV_THREAD：内核创建新线程，调用 sigev_notify_function
    • SIGEV_THREAD_ID（Linux 特有）：发送 signal 给指定线程

    🔹 signal() 和 sigaction() 是“信号的处理机制”

    这两个函数用于定义收到某个 signal 时如何响应，比如 SIGINT（Ctrl+C）或 SIGTERM 等。
