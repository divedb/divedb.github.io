UNQUALIFIED NAMES

  • Unqualified names undergo a two step process: unqualified lookup and “maybe” ADL
  • Let U be the set produced by (U)nqualified lookup and A the set produced by (A)DL
  • If the set U contains any of the following:
    • A declaration of a class member
    • A block scope function declaration that is not a using declaration
    • A declaration that is neither a function nor a function template

Then A is empty and the lookup does not proceed to step 2.

STEP 2: ADL

For each argument type T in a function call there are two sets (each of size zero or more)

  • Set E (assoicated entities)
  • Set N (associated namespaces)

typedefs and using-declarations do not! contribute to this set.

  • T is a fundamental type.

  • T is of class type.

    • E is [class itself, the class which is a member to (if any) and any indirect or direct base classes]

    • N is [innermost enclosing namespace of its associated entities]

      • Step 1. Unqualified lookup finds nothing U is empty.
      • Step 2. E(pipe, user_pipe) and N[lib, user]. ADL finds [lib::connect, user::connect]
      • Overload resolution picks user::connect() as a better match
  1. T is an enum
    • E is [the enum itself, if it’s a class member: the member’s class]
    • N is [innermost enclosing namespaces of its declaration]

  1. T is a pointer to X or an array of X.

    N,E is [entities associated with X]

    • Unqualified lookup finds nothing. U is empty.
    • ADL finds, E(lib::pipe), N[lib]. Therefore, we feed. [lib::connect(pipe* ptr)] to the overload resolution

  2. T is a function type

    N, E is [function parameter types, return type]

    • Unqualified lookup finds nothing. U is empty.
    • ADL finds, E[lib::pipe], N[lib]. Therefore, A is [lib::process()]
