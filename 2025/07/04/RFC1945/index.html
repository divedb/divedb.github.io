    RFC1945

    发表于 2025-07-04 更新于 2025-07-07 分类于 网络

    摘要

    超文本传输协议（HTTP）是一种应用层协议，具有轻量级和高效的特性，适用于分布式、协作式的超媒体信息系统。HTTP是一种通用的、无状态的、面向对象的协议，通过扩展其请求方法（命令），可以应用于多种场景，例如名称服务器和分布式对象管理系统等。HTTP的一个重要特性是其对数据表示类型的明确标识，这使得数据传输的两端可以独立实现，无需关心对方的具体数据格式。

    💡HTTP协议中，数据是有“类型”的，例如Content-Type: application/json，告诉接收方数据是JSON、HTML还是图片。

    自1990年以来，HTTP已被全球范围的World-Wide Web（万维网）信息系统广泛采用。本规范描述 HTTP/1.0协议的通用使用方式。

    Introduction

    Purpose

    超文本传输协议（HTTP）是一种应用层协议，专为分布式协作式超媒体信息系统设计，具有轻量高效的特点。
    自1990年以来，HTTP已被全球范围的万维网（World-Wide Web）信息系统广泛采用。本规范描述了被广泛称为HTTP/1.0的通用实现标准，重点阐述了大多数HTTP/1.0客户端和服务器普遍支持的功能特性。

    本规范分为两部分：

    • 文档主体部分描述了在大多数实现中表现一致的HTTP功能。
    • 附录D则列出了那些实现较少或存在不一致的功能。

    在实际应用中，信息系统不仅需要简单的数据检索，还需要更多功能，例如搜索、前端更新和批注。HTTP通过可扩展的方法机制（Method）表示请求的意图，并基于统一资源标识符（URI）的引用规范，通过资源定位符（URL）或资源名称（URN来指定方法操作的目标资源。其消息格式借鉴了互联网邮件和多用途互联网邮件扩展（MIME）的成熟方案。

    此外，HTTP还作为通用协议，协调用户代理与代理/网关之间的通信，可对接SMTP、NNTP、FTP、Gopher和WAIS等多种互联网协议。这种设计既实现了跨应用超媒体资源访问，又显著简化了用户代理的实现复杂度。

    术语

    本规范中使用了若干术语，用于描述HTTP通信中的参与方角色以及通信对象：

    术语 定义
    connection 建立于两个应用程序之间的传输层虚拟电路，专用于通信交互。
    message HTTP通信的基本单元，由的八位组结构化序列构成。
    request HTTP请求消息。
    response HTTP响应消息。
    resource 通过URI标识的网络数据对象或服务。
    entity 数据资源的某种具体表现形式，或服务资源的某次响应表现，可包含在请求或响应消息中。 一个实体包含两部分。实体头（entity headers）：提供关于实体的元信息； 实体主体（entity body）：携带实际的数据内容。
    client 用于建立连接并发送请求的应用程序。
    user agent 发起请求的客户端，通常是浏览器、编辑器、爬虫、或其他面向用户的工具。
    server 用于接收连接并处理请求、返回响应的应用程序。
    origin server 资源实际存储或即将被创建的服务器。
    proxy 一种中间程序，既作为服务器接收请求，又作为客户端代表其他客户端发起请求。 代理可以内部直接处理请求，或将请求转发给其他服务器（在转发时可能对请求进行转换）。 代理必须能够解析请求消息，并在必要时对请求内容进行重写再转发。代理常被用作客户端与防火墙之间的网关； 为不支持某些协议的用户代理处理请求的辅助程序。
    gateway 一种为其他服务器充当中介的服务器。 与代理不同，网关对客户端表现得像是请求资源的源服务器，客户端通常不知道自己正在通过网关通信。网关常被用作服务器端与防火墙之间的网关； 协议转换器，将HTTP请求转发给非HTTP系统上的资源。
    tunnel 一种中间程序，仅作为两个连接之间的透明中继。 隧道建立后，不参与HTTP通信的任何内容解析，尽管它可能是由HTTP请求启动的。 当两端连接关闭时，隧道也随之终止。隧道常用于以下场景： 必须经过某个中介程序，但该程序无法或不应解析通信内容时。 比如：HTTPS通过HTTP代理隧道传输。
    cache 程序本地存储的响应消息，以及管理这些消息的存储、检索和删除的子系统。缓存的作用是存储可缓存的响应，以便在后续相同请求时减少响应时间和网络带宽消耗。

    同一个程序可能既是客户端，也可以是服务器；本文中的术语是描述在当前连接中扮演的角色，而不是指程序的全部功能。同样，服务器在不同请求中可能分别扮演：

    • 源服务器
    • 代理
    • 网关
    • 隧道
    总体操作流程

    HTTP协议基于请求/响应模型。客户端与服务器建立连接后，向服务器发送请求。请求由：

    • 请求方法、URI和协议版本。
    • 后面跟着类似MIME格式的消息体，其中包含请求的附加信息、客户端信息。
    • 以及可能存在的请求内容。

    服务器则以状态行（status line）作为响应头，其中包含：

    • 协议版本和表示成功或失败的状态码。
    • 随后是类似MIME格式的消息体，包含服务器信息、实体的元数据。
    • 以及可能的响应内容。

    • 代理：充当转发代理，接收包含完整URI的请求，可对请求的部分或全部内容进行重写，然后将修改后的请求转发给目标 URI所指向的服务器。
    • 网关：作为上层接收方，负责与其他服务器进行交互，并在必要时将请求转换为底层服务器所使用的协议。
    • 隧道：作为两个连接之间的中继，不会更改消息内容。当通信必须经过某个无法理解消息内容的中间设备（如防火墙）时，常使用隧道。

    在互联网环境下，HTTP通常基于TCP/IP连接进行通信，默认端口为TCP 80，但也可使用其他端口。不过，这并不意味着 HTTP只能运行在TCP/IP之上，它同样可以基于其他可靠传输协议，只要这些协议能保证可靠传输即可。HTTP/1.0请求和响应如何映射到底层传输协议的数据单元，不属于本规范的讨论范围。

    除了实验性应用之外，目前通用做法是：每次请求前由客户端建立连接，服务器在发送完响应后关闭连接。客户端和服务器都应做好连接被提前关闭的准备，无论是由于用户操作、自动超时，还是程序故障导致的关闭，都应当以可预期的方式进行处理。无论是哪一方关闭连接，一旦连接断开，当前请求即被视为终止，无论其状态如何。

    HTTP与MIME

    HTTP/1.0使用了许多MIME协议（定义于RFC 1521）中的结构。附录C说明了HTTP使用互联网媒体类型（Internet Media Types）与互联网电子邮件（Internet mail）使用方式的差异，并阐释了这些差异的原因。

    记法约定与通用语法

    增强型BNF

    本规范中的所有机制，既采用文字说明，也采用增强型巴科斯-诺尔范式（BNF）表示，这种表示方式类似于RFC 822。实现者需要熟悉这种记法，才能准确理解规范内容。增强型BNF包含以下构造方式：

    ✅ name = definition

    规则名称直接写出（不包含尖括号<和>），通过等号=与其定义部分分隔。空白字符（whitespace）通常不具备语法意义，但当规则定义跨多行书写时，应通过缩进来表示行的延续。部分基础规则采用全大写形式，如SP、LWS、HT、CRLF、DIGIT、ALPHA等。定义中在需要帮助辨识的地方会使用尖括号将规则名称括起来，以提高可读性。

    假设语法规则很长，可以这样写：

    long-rule = element1 element2 element3 element4
    element5 element6 element7

    ✅ literal

    双引号" "用于表示字面文本（literal text）。除非另有说明，⚠️字面文本的匹配不区分大小写。

    ✅ rule1 | rule2

    规则之间用竖线 | 分隔，表示可选的替代项。例如，yes | no表示可以匹配”yes”或”no”。

    ✅ (rule1 rule2)

    括号()用于将多个元素组合为一个整体。例如，(elem (foo | bar) elem)表示可以匹配elem foo elem或elem bar elem。

    ✅ *rule

    星号*表示重复出现，完整形式为<n>*<m>element，表示元素出现次数在n到m次之间。常见情况如下：

    • *(element)：可重复任意次数，包括0次（默认0次到无限次）；
    • 1*element：至少出现1次；
    • 1*2element：出现1或2次。

    ✅ [rule]

    方括号[]表示可选内容，[foo bar]等价于*1(foo bar)，表示出现0或1次。

    ✅ N rule

    具体指定重复次数，<n>(element)表示恰好n次重复，等价于<n>*<n>(element)。例如：

    • 2DIGIT表示一个由两位数字组成的数字；
    • 3ALPHA表示由3个字母组成的字符串。

    ✅ #rule

    井号#定义以逗号分隔的元素列表，语法类似于*，完整形式为<n>#<m>element，表示由n到m个元素组成，元素之间用一个或多个逗号 , 和可选的线性空白符（LWS）分隔。这样，原本复杂的列表规则：

    • 允许列表中存在空元素，但空元素不计入总数量。
      例如，(element), , (element)被视为两个元素，而非三个。
    • 如果要求至少有一个元素，那么必须包含至少一个非空元素。

    默认值为0和无限，因此：

    • #(element)：允许任意数量，包括0个；
    • 1#element：至少1个；
    • 1#2element：1个或2个。

    ✅ ; comment

    分号;通常与规则主体有一定间隔，用于引入注释，注释内容持续到行尾。这是在规范中并列加入说明性注释的简单方式。

    ✅ 隐含的*LWS

    本规范描述的语法以单词为单位。除非特别说明，相邻的单词（token或quoted-string）之间、token与特殊分隔符（tspecials）之间，都可以包含线性空白符（LWS），不会改变语义。

    不过，任意两个token之间必须至少有一个分隔符（tspecials），否则会被解析为单一token。

    尽管如此，应用程序在生成HTTP内容时，仍应尽可能遵循通用格式，因为有些实现可能无法正确处理非标准格式。

    协议参数

    HTTP版本

    HTTP协议采用「<主版本号>.<次版本号>」的方式表示协议版本。这种版本机制主要用于表明发送方所采用的消息格式及其对之后HTTP通信的理解能力，而不是表示具体实现了哪些功能。

    🔍 举例说明

    • 如果客户端发送的是HTTP/1.0，它告诉服务器：
      • “我发的请求是1.0格式”；
      • “我理解1.0的协议内容”；
      • 并不是说“我支持缓存”或“我支持持久连接”这种功能级别的声明。

    如果新增的消息组件不改变协议的通信行为，或者只是扩展了某些字段的取值范围，则无需修改版本号。只有在新增功能不会改变消息的基本解析算法，但可能影响消息语义、表明发送方具备额外能力时，才会增加次版本号（minor）。而当协议中的消息格式发生更改时，才会增加主版本号（major）。

    HTTP消息的版本号通过消息首行中的HTTP-Version字段表示。如果首行未声明协议版本，接收方必须假定该消息采用最简单的HTTP/0.9格式。

    请注意，主版本号和次版本号是独立的整数，且每个部分都可能超过一位数字。因此，HTTP/2.4小于HTTP/2.13，后者又小于HTTP/12.3。接收方在解析时应忽略版本号前的零，发送方不应生成带前导零的版本号。本规范定义了HTTP协议的0.9和1.0两个版本。凡是发送完整请求（Full-Request）或完整响应（Full-Response）的应用，必须在消息中声明版本为 HTTP/1.0。

    HTTP/1.0服务器必须遵循以下要求：

    • 能够识别HTTP/0.9和HTTP/1.0请求的Request-Line格式；
    • 能够理解符合HTTP/0.9或HTTP/1.0格式的任何合法请求；
    • 必须以客户端请求所用的协议版本格式正确响应。

    HTTP/1.0客户端必须遵循以下要求：

    • 能够识别HTTP/1.0响应的Status-Line格式；
    • 能够理解符合HTTP/0.9或HTTP/1.0格式的任何合法响应。

    代理和网关应用在转发请求时，必须特别注意请求的版本号是否高于自身支持的协议版本。由于版本号表明了发送方的协议能力，代理或网关不得发送高于自身支持版本号的消息。如果收到高版本的请求，代理/网关必须：

    • 要么将请求的版本号降级；
    • 要么直接返回错误响应。

    对于版本号低于自身的请求，代理/网关可以选择将版本号升级再转发；但无论如何，其对该请求的响应必须遵循前文所述服务器行为的要求。

    统一资源标识符（URI）

    URI曾有多种名称，包括WWW地址、通用文档标识符、通用资源标识符，以及后来结合的统一资源定位符（URL）和统一资源名称（URN）。

    从HTTP的角度看，统一资源标识符只是一些特定格式的字符串，用于通过名称、位置或其他属性来标识一个网络资源。

    通用语法

    在HTTP中，URI可以是：

    • 绝对形式（absolute form），或
    • 相对于某个已知基准URI的相对形式（relative form）。

    两种形式的区别在于：绝对URI总是以「方案名（scheme）」加冒号（:）开头。

    语法定义如下：

    📜 语法说明

    • absoluteURI：如http://example.com/index.html。
    • relativeURI：如/path/to/resource，其含义依赖于上下文中的基准URI。
    • escape：百分号编码，如%20表示空格。
    • reserved / safe / unsafe：对字符的分类，决定是否需要转义。

    更完整的URL语法和语义，请参考RFC 1738和RFC 1808。

    需要注意的是：

    • 上述BNF语法包含了一些RFC 1738中不允许出现在URL中的「national」字符；
    • 这是因为HTTP服务器对于地址中rel_path部分的未保留字符（unreserved characters）并不做严格限制；
    • 同时，HTTP代理可能会收到包含非标准URI的请求，这些URI不一定符合RFC 1738的规范。

    日期/时间格式

    HTTP/1.0应用在历史上支持三种不同的日期/时间表示格式：

    • 第一种格式是互联网标准，属于RFC 1123的固定子集（RFC 1123是对RFC 822的更新），推荐使用；
    • 第二种格式虽然广泛使用，但基于已废弃的RFC 850，且年份只有两位；
    • HTTP/1.0的客户端和服务器在解析日期时，应该接受上述三种格式，但不应当生成第三种（asctime）格式。

    📌 注意：
    接收端应尽可能兼容来自非HTTP应用生成的日期值，例如通过代理或网关转发的SMTP、NNTP消息中可能包含这些日期格式。

    ✅ 时区要求

    所有HTTP/1.0的日期/时间必须采用协调世界时 (UT)，也称为格林尼治标准时间 (GMT)，不能使用其他时区。在前两种格式中，使用GMT标识时区；第三种格式未明确指定时区，但应视为GMT。

    在 HTTP 协议中，Host 头部的存在是为了解决一个关键问题：单台服务器托管多个域名/网站（虚拟主机）时的请求路由。尽管请求行（Request Line）中包含 URL 路径，但仅有路径信息无法满足现代 Web 服务的需求。以下是详细解释：

    1. 请求行（Request Line）的局限性

    原始的 HTTP/1.0 请求例如：

    text

    复制

    下载

    • 问题：当一台服务器托管 example.com 和 test.com 时，仅凭路径 /index.html 无法区分用户想访问哪个域名下的资源。

    2. Host 头部的作用

    HTTP/1.1 强制要求 Host 头部：

    text

    复制

    下载

    • 核心功能：告诉服务器客户端请求的是哪个域名。
    • 必要性：
      • 支持虚拟主机（单 IP 多网站）
      • 实现 CDN 精准调度
      • 处理 HTTPS 的 SNI（Server Name Indication）

    3. 关键场景分析

    (1) 虚拟主机（Virtual Host）

    • 物理架构：
      单台服务器（如 IP 192.0.2.1）同时托管：
      • example.com
      • test.com
    • 无 Host 头部时：
      服务器无法区分请求该返回哪个网站的内容。

    (2) HTTPS 加密（SNI 扩展）

    • TLS 握手阶段：
      客户端通过 Host 告诉服务器该发送哪个域名的证书（在加密前完成域名匹配）。

    (3) 反向代理/负载均衡

    • 代理服务器：
      根据 Host: api.service.com 和 Host: web.service.com 将流量路由到不同后端集群。

    4. 技术对比

    组件 信息类型 示例 作用域
    请求行（URL路径） 资源路径 GET /index.html HTTP/1.1 仅定位服务器内资源
    Host 头部 目标域名 Host: example.com 定位虚拟主机
    IP 地址 物理服务器位置 192.0.2.1:80 网络层寻址

    如果 HTTP/1.1 请求缺失 Host 头部：

    • 服务器返回 400 Bad Request
      （HTTP/1.1 规范强制要求）

    • 示例错误：

      text

      复制

      下载

    6. 现代扩展应用

    • HTTP/2 和 HTTP/3：
      通过 :authority 伪头部字段继承 Host 的功能。
    • 云原生场景：
      Kubernetes Ingress 根据 Host 路由到不同 Service。

    总结

    • 历史原因：HTTP/1.1 引入 Host 解决多域名托管问题。
    • 核心价值：解耦「物理 IP」和「逻辑域名」，实现灵活的 Web 架构。
    • 不可替代性：即使请求行包含路径，Host 仍是现代互联网基础设施（CDN/HTTPS/微服务）的基石。
