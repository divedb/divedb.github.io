mount的时候将fd和挂载点的信息传递给内核。

✅ MS_RDONLY

  • 含义：将文件系统挂载为只读。
  • 影响：你不能对该文件系统写入文件或更改元数据。
  • 常用于：挂载 CD-ROM、ISO 镜像、或者只读模式修复文件系统。

✅ MS_NOSUID

  • 含义：挂载后该文件系统内的所有文件即使有 setuid（SUID）或 setgid（SGID）位，也会被忽略。
  • 影响：
    • 即使程序文件设置了 SUID（比如 /usr/bin/passwd），执行时也不会提升权限。
  • 用途：
    • 挂载 USB、SD 卡、网络文件系统时，防止恶意程序借助 SUID/SGID 提权。

✅ MS_NODEV

  • 含义：挂载后该文件系统中的设备 special 文件（即字符设备文件和块设备文件）无法使用。
  • 影响：即使你创建了 /dev/sda 这样的设备文件，程序也不能通过它访问设备。
  • 用途：
    • 安全限制：挂载某些不可信来源的文件系统（如 U 盘、loop 文件），防止通过 fake device file 做非法操作。

在 FUSE（Filesystem in Userspace）中，struct mount_opts 用于传递挂载参数，控制文件系统的行为。以下是各字段的详细说明：

字段含义及作用

1. 权限控制

字段 类型 说明
allow_other int 是否允许非挂载用户访问文件系统（默认仅挂载用户可访问）。 需配合 /etc/fuse.conf 中的 user_allow_other 选项使用。
allow_root int 是否允许 root 用户访问文件系统（默认允许）。 若禁用，root 用户将无法访问。

2. 挂载行为控制

字段 类型 说明
nonempty int 是否允许挂载到非空目录（默认禁止）。 启用后，原有目录内容会被隐藏，直到卸载。
blkdev int 是否挂载的是块设备（而非普通文件系统）。 用于特殊场景（如直接挂载磁盘镜像）。

3. 辅助标志

字段 类型 说明
ishelp int 是否显示帮助信息（由 -h/--help 触发）。
flags int 挂载的全局标志位，例如： - MS_RDONLY（只读挂载） - MS_NOSUID（忽略 SUID 权限）。

4. 标识与分类

字段 类型 说明
fsname char* 文件系统的名称标识（通过 -o fsname=NAME 指定）。 用于 df 或 /proc/mounts 中显示。
subtype char* 文件系统的子类型（如 fuse.sshfs）。 影响 /proc/mounts 中的显示格式（type.subtype）。
subtype_opt char* 子类型的挂载选项（内部使用）。

5. 选项传递

字段 类型 说明
mtab_opts char* 写入 /etc/mtab 的选项字符串。
fusermount_opts char* 传递给 fusermount 工具的选项。
kernel_opts char* 传递给内核的原始挂载选项（未解析的 -o 参数）。

总结

  • 权限控制：allow_other 和 allow_root 决定谁可以访问文件系统。
  • 挂载行为：nonempty 和 blkdev 影响挂载的灵活性。
  • 标识与分类：fsname 和 subtype 用于显示和管理。
  • 选项传递：通过 mtab_opts 等字段将参数分发给不同组件。

你遇到这个错误：

这是 Linux 中常见的错误，说明你挂载在 /tmp/mnt 上的文件系统（可能是 FUSE 文件系统，比如 sshfs、rclone、你自己写的 libfuse 项目等）已经失联或崩溃了，但挂载点还存在。

✅ 错误含义

  • “Transport endpoint is not connected“ 指的是内核认为挂载的文件系统（通常是 FUSE 类型）底层连接已断开。
  • 比如 sshfs 中 ssh 链接断了；libfuse 中 mount 的服务端进程崩溃了或退出了。

1️⃣ 首选：卸载挂载点

说明：

  • -l 表示 lazy umount，即立即断开 /tmp/mnt，底层等进程释放后再完全清理。
  • 这是处理 FUSE 挂载挂死问题最推荐的方式。

🔁 总结命令清单

要看懂 libfuse 的源码（尤其是 FUSE 2.x，例如你用的 2.8.0），建议从使用者的视角出发，再深入实现层。以下是阅读建议路径，分阶段推荐关键文件和内容：

🧭 第一阶段：理解外部接口（你程序调用的是哪部分）

✅ 推荐先看：

文件 内容
include/fuse.h 最主要的用户 API，定义了你实现的 fuse_operations 结构体
example/hello.c 官方示例，说明 FUSE 程序如何初始化和运行
lib/fuse.c 实现了 fuse_main 和核心绑定逻辑，连接用户提供的操作函数和内核交互

🧭 第二阶段：理解事件循环与会话管理

✅ 推荐阅读：

文件 内容
lib/fuse_loop.c fuse_loop() 的实现，主事件循环：收请求 / 调操作 / 回响应
lib/fuse_session.c fuse_session_new()、fuse_session_process()，封装一个“挂载会话”
lib/fuse_signals.c 处理 SIGINT/SIGTERM 等退出信号
lib/fuse_opt.c 参数解析，例如 fuse_main() 中传入的 argv 如何转为配置

🧭 第三阶段：深入理解与内核通信的细节

✅ 推荐阅读：

文件 内容
lib/fuse_kern_chan.c 管理 /dev/fuse 的读写，对应 FUSE 内核接口
lib/fuse_lowlevel.c “底层接口”，直接处理 FUSE 协议请求（用于实现 CUSE、精细控制）
include/fuse_kernel.h FUSE 协议结构体定义（例如 fuse_in_header），与内核通信的数据结构
lib/mount.c + lib/mount_util.c 挂载相关逻辑，底层执行 fusermount 命令或直接 syscall

🔎 补充（可选）

  • lib/helper.c：一些小工具辅助函数
  • lib/cuse_lowlevel.c：字符设备 in userspace（如果你要研究 CUSE）
  • lib/ulockmgr.c：用户态锁管理器（较少用）

✅ 阅读建议顺序总结

阶段 目标 文件
🟩 1. 外部接口 搞清楚你代码调用了什么 include/fuse.h, example/hello.c, lib/fuse.c
🟨 2. 会话和循环 理解 FUSE 是如何调你提供的函数的 lib/fuse_loop.c, lib/fuse_session.c, lib/fuse_opt.c
🟥 3. 内核交互 真正和 /dev/fuse 通信细节 lib/fuse_kern_chan.c, include/fuse_kernel.h
🧩 4. 其他 mount 辅助工具、低层处理 lib/mount.c, lib/fuse_lowlevel.c 等
