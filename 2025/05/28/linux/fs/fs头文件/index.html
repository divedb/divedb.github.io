    fs头文件

    发表于 2025-05-28 更新于 2025-07-02 分类于 Linux

    🧩概述

    Linux内核中的include/linux/fs.h是文件系统核心头文件，定义了很多用于VFS（虚拟文件系统）的重要数据结构。理解这些结构是掌握Linux文件系统实现的基础。

    📚 数据结构

    🎯struct super_block`

    🌟描述：表示一个挂载的文件系统（超级块），描述整个文件系统的全局信息。

    🌱部分字段：

    🎯struct inode`

    🌟描述：表示一个索引节点（即文件或目录的元数据）。

    🌱部分字段：

    🎯struct file_operations`

    🌟描述：是字符设备、块设备和文件系统操作的核心结构，定义该设备或文件的各种操作函数指针，是虚拟文件系统层与驱动之间的桥梁。

    🌱部分字段：

    🎯file

    🌟描述：代表一个进程打开的某个文件实例，它并不仅限于设备驱动程序。系统中每个打开的文件在内核空间都有一个对应的file结构。

    🌱部分字段：

    🎯dentry

    🌟描述：dentry（directory entry）是Linux内核中的目录项缓存结构，用于加速路径查找。例如：/home/user/file 中的home、user、file）在VFS层都有对应的dentry实例。它连接了文件系统的层级结构，并缓存了路径与inode的映射关系，是文件系统查找的核心加速机制。

    🌱部分字段：

    🎯address_space

    🌟描述：address_space是Linux内核中用于管理文件数据缓存的核心数据结构，它抽象了文件的页缓存（page cache）映射关系。每个文件的内容都通过address_space结构与内存中的页面（struct page）关联，实现高效的文件数据读写缓存和同步。

    🌱部分字段：

    nameidata 是 Linux 内核中用于 路径名解析（pathname resolution） 的核心结构体。它在内核 VFS 层承担着路径解析过程中维护状态、传递上下文、控制解析流程的作用。你可以理解它是 open, stat, unlink 等系统调用背后路径处理的“导航器”。

    📝FAQ

    操作系统如何知道inode 3的1~1024字节已经缓存？

    Linux的页缓存是以页为单位缓存文件内容的，也就是每4KB为一页（假设页大小是4KB），1~1024字节对应的是第0页。

    查找过程：

    1. inode -> address_space：
      • 每个inode都包含一个address_space成员（inode->i_mapping），这就是该文件的页缓存的入口。
    2. address_space -> i_pages：
      • i_mapping->i_pages 是一个xarray，是页缓存的核心存储结构。
      • 它用page index（通常是文件偏移 / PAGE_SIZE）作为key。
      • 所以如果你要读第0页（offset 0 ~ 4095字节），内核会在i_pages里查key=0的页。
    3. 是否命中：
      • 如果xarray中已有key=0的 struct page*，说明该页在页缓存中，直接返回。
      • 如果没有命中，就调用a_ops->readpage()，让底层文件系统读入页并插入i_pages。
    address_space是如何组织的？是链表还是红黑树？
    • 每个address_space结构体管理的是一个文件的页缓存。
    • 各个页（struct page*）以page index为键存放在xarray（i_pages）中。
    • xarray是一种多层索引树结构，查找非常快，支持稀疏索引。

    ➡️ 所以不是链表也不是红黑树，是xarray。：

    • 每个inode（唯一对应一个文件）就有一个address_space
    • 内核操作page cache时只关心特定文件，对应特定inode -> mapping，根本不需要遍历全部
    address_space的应用场景？
    1. 普通文件读写（read/write）
    • 调用read()或write()时，最终通过inode->i_mapping访问页缓存（address_space）。
    • 如果没有命中页缓存，会触发page fault，由a_ops->readpage()填充页缓存。
    1. mmap内存映射文件

    内存映射文件是address_space的重要使用场景之一。

    mmap时：

    • 创建一个vm_area_struct（VMA）并关联到file->f_mapping，即 inode->i_mapping。
    • 每次访问映射区（触发page fault）时，走filemap_fault()，通过address_space->i_pages查找/加载页。

    作用：

    • mmap共享文件时，所有进程的VMA映射都指向同一个address_space。
    • 保证多个进程访问相同映射区域时，共享页缓存，避免重复I/O。

    再经由file->f_mapping，使用其page cache。

    1. 页写回（Writeback）与脏页回收
    • 脏页由mark_page_dirty()标记，存入address_space。
    • 回写时调用a_ops->writepage()等函数。
    • 所有脏页都由address_space管理。
    1. Direct I/O与Page Cache协调
    • 虽然Direct I/O绕过页缓存，但文件的元数据（如大小）仍然依赖inode和address_space。
    • 写入前必须确保address_space中的脏页被flush，否则数据不一致。
    1. 文件截断（truncate）、hole punch（打洞）
    • 截断文件时（如ftruncate()），需要从address_space->i_pages中移除相关页。
    • 释放内存、更新inode信息。
    1. 写时复制（COW）机制支持
    • 多个进程mmap同一个文件时，address_space是页共享的桥梁。
    • 写入时触发写时复制，更新address_space页表，避免数据冲突。

    📎 特殊说明：shmem和tmpfs的address_space

    不仅磁盘文件使用address_space，内存文件系统如tmpfs也使用它：

    • tmpfs文件数据页也挂在i_mapping->i_pages上。
    • 例如/dev/shm/、匿名共享内存页。

    🧠 使用场景

    generic_fillattr 通常在文件系统未提供自定义的 getattr 操作时被调用，作为默认的属性填充函数。它确保了即使在没有特定实现的情况下，文件系统也能提供基本的文件属性信息。

    🔄 与 vfs_getattr 的关系

    vfs_getattr 是一个更高级别的接口，用于获取文件属性信息。它可能会调用文件系统提供的 getattr 操作，如果未定义，则退回到使用 generic_fillattr。

    2. 主要接口函数

    文件操作相关

    • int vfs_read(struct file *file, char __user *buf, size_t count, loff_t *pos)
      从文件中读取数据。
    • int vfs_write(struct file *file, const char __user *buf, size_t count, loff_t *pos)
      向文件写入数据。
    • int vfs_open(struct inode *inode, struct file *file)
      打开文件。
    • int vfs_close(struct file *file)
      关闭文件。
    • int vfs_fsync(struct file *file, loff_t start, loff_t end, int datasync)
      同步文件数据到磁盘。
    • int vfs_truncate(const struct path *path, loff_t length)
      截断文件。

    路径和目录操作

    • int vfs_mkdir(struct inode *dir, struct dentry *dentry, umode_t mode)
      创建目录。

    • int vfs_rmdir(struct inode *dir, struct dentry *dentry)
      删除目录。

    • int vfs_unlink(struct inode *dir, struct dentry *dentry)
      删除文件。

    • struct dentry *lookup_one_len(const char *name, struct dentry *parent, int len)
      查找目录项。

    3. 文件系统注册和管理

    • int register_filesystem(struct file_system_type *fs_type)
      注册文件系统类型。
    • void unregister_filesystem(struct file_system_type *fs_type)
      注销文件系统。
    • struct file_system_type
      文件系统类型结构体，定义文件系统的名字和挂载回调等。

    4. 其他重要接口

    • struct file *file_open(const char *filename, int flags, umode_t mode)
      打开文件（更底层，内核内部使用）。
    • void file_close(struct file *file)
      关闭文件。
    • ssize_t kernel_read(struct file *file, void *buf, size_t count, loff_t *pos)
      内核态读取文件。
    • ssize_t kernel_write(struct file *file, const void *buf, size_t count, loff_t *pos)
      内核态写文件。

    5. 典型用法场景

    • 文件操作：内核模块中访问文件时调用 kernel_read、kernel_write。
    • 文件系统开发：实现自己的文件系统时，使用 register_filesystem 注册。
    • 目录管理：操作目录项，调用 vfs_mkdir、vfs_unlink 等。
