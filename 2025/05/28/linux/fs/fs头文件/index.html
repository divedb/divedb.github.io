    📘 fs.h常见数据结构

    ​ Linux内核中的include/linux/fs.h是文件系统相关的核心头文件，定义了众多用于VFS（虚拟文件系统）的重要数据结构。理解这些结构是掌握Linux文件系统实现的基础。

    2. 主要接口函数

    文件操作相关

    • int vfs_read(struct file *file, char __user *buf, size_t count, loff_t *pos)
      从文件中读取数据。
    • int vfs_write(struct file *file, const char __user *buf, size_t count, loff_t *pos)
      向文件写入数据。
    • int vfs_open(struct inode *inode, struct file *file)
      打开文件。
    • int vfs_close(struct file *file)
      关闭文件。
    • int vfs_fsync(struct file *file, loff_t start, loff_t end, int datasync)
      同步文件数据到磁盘。
    • int vfs_truncate(const struct path *path, loff_t length)
      截断文件。

    路径和目录操作

    • int vfs_mkdir(struct inode *dir, struct dentry *dentry, umode_t mode)
      创建目录。

    • int vfs_rmdir(struct inode *dir, struct dentry *dentry)
      删除目录。

    • int vfs_unlink(struct inode *dir, struct dentry *dentry)
      删除文件。

    • struct dentry *lookup_one_len(const char *name, struct dentry *parent, int len)
      查找目录项。

    3. 文件系统注册和管理

    • int register_filesystem(struct file_system_type *fs_type)
      注册文件系统类型。
    • void unregister_filesystem(struct file_system_type *fs_type)
      注销文件系统。
    • struct file_system_type
      文件系统类型结构体，定义文件系统的名字和挂载回调等。

    4. 其他重要接口

    • struct file *file_open(const char *filename, int flags, umode_t mode)
      打开文件（更底层，内核内部使用）。
    • void file_close(struct file *file)
      关闭文件。
    • ssize_t kernel_read(struct file *file, void *buf, size_t count, loff_t *pos)
      内核态读取文件。
    • ssize_t kernel_write(struct file *file, const void *buf, size_t count, loff_t *pos)
      内核态写文件。

    5. 典型用法场景

    • 文件操作：内核模块中访问文件时调用 kernel_read、kernel_write。
    • 文件系统开发：实现自己的文件系统时，使用 register_filesystem 注册。
    • 目录管理：操作目录项，调用 vfs_mkdir、vfs_unlink 等。

    ✅ 核心结构体字段解释

    字段名 类型 含义
    host struct inode* 关联的 inode（文件）或块设备。即这个页缓存属于哪个文件/设备。
    i_pages struct xarray 缓存页的容器（xarray 替代了老的 radix tree），存储所有缓存页 struct page。
    invalidate_lock struct rw_semaphore 读写锁，用于在无效化页缓存时防止竞态，如 truncation/invalidation 操作中保护 page cache 与 block mapping 的一致性。
    gfp_mask gfp_t 内存分配标志（Get Free Page flags），控制如何为页缓存分配内存（如是否允许等待、能否回收、可用 zone 等）。
    i_mmap_writable atomic_t 被 mmap 且具有写权限（VM_SHARED + VM_MAYWRITE）映射的次数。影响 page write-back 权限判断。
    nr_thps atomic_t (仅非 shmem 文件) page cache 中 huge page (THP, Transparent Huge Pages) 的数量，仅在启用 CONFIG_READ_ONLY_THP_FOR_FS 时存在。
    i_mmap struct rb_root_cached 表示该文件所有的 mmap 映射区（VMA）的红黑树，用于追踪文件到内存的所有映射。
    nrpages unsigned long 当前页缓存中页的数量。受 i_pages 的内部锁保护。
    writeback_index pgoff_t 回写（writeback）从这个页索引位置开始，配合 a_ops->writepages() 使用。
    a_ops const struct address_space_operations* 操作函数集合（如 readpage, writepage, invalidatepage 等），提供页缓存与底层文件系统之间的接口。
    flags unsigned long address_space 的标志位（如是否 dirty、是否需要回写等），定义为 AS_* 宏。
    wb_err errseq_t 最近一次回写的错误，用于用户态通过 fsync 等方式获取错误状态。
    i_private_lock spinlock_t 自旋锁，供文件系统或块设备驱动使用，保护以下几个字段。
    i_private_list struct list_head 供文件系统私用的链表头，比如 writeback 的 pending 列表等。
    i_mmap_rwsem struct rw_semaphore 读写锁，保护 i_mmap 红黑树和 i_mmap_writable，确保 mmap 映射在添加、删除时是安全的。
    i_private_data void* 文件系统专用的私有数据指针，例如用于保存 journaling 信息等。

    🌟 一、操作系统如何知道 inode 3 的 1~1024 字节已经缓存？

    Linux 的页缓存是以页为单位缓存文件内容的，也就是每 4KB 为一页（假设页大小是 4KB），你的这个范围（1~1024 字节）对应的是第 0 页。

    查找过程：

    1. inode -> address_space：
      • 每个 inode 都包含一个 address_space 成员（inode->i_mapping），这就是该文件的页缓存的入口。
    2. address_space -> i_pages：
      • i_mapping->i_pages 是一个 xarray（以前是 radix tree），是页缓存的核心存储结构。
      • 它用 page index（通常是文件偏移 / PAGE_SIZE）作为 key。
      • 所以如果你要读第 0 页（offset 0 ~ 4095 字节），内核会在 i_pages 里查 key=0 的页。
    3. 是否命中：
      • 如果 xarray 中已有 key=0 的 struct page*，说明该页在页缓存中，直接返回。
      • 如果没有命中，就调用 a_ops->readpage()，让底层文件系统读入页并插入 i_pages。

    🌲 二、address_space 是如何组织的？是链表还是红黑树？

    答案：

    • 每个 address_space 结构体管理的是一个 文件的页缓存。
    • 各个页（struct page*）以 page index 为键存放在 xarray（i_pages） 中。
    • xarray 是一种 多层索引树结构（取代了 radix tree），查找非常快，支持稀疏索引。

    ➡️ 所以不是链表也不是红黑树，是 xarray。

    你问的“多个 address_space 是不是组成链表或红黑树”，实际上内核没有全局组织 address_space 的链表，因为：

    • 每个 inode（唯一对应一个文件）就有一个 address_space
    • 内核操作 page cache 时只关心特定文件，对应特定 inode -> mapping，根本不需要遍历全部

    🌱 三、address_space 的根节点（入口）存储在哪里？是在 task_struct 中吗？

    不是的，不在 task_struct 中。

    正确结构：

    • 在 Linux 中，每个 struct inode 都有一个成员：

    • 这就是页缓存的入口，每个 inode 只有一个 address_space。

    • 所以：

      • 入口根节点是 inode->i_mapping
      • 不在进程结构 task_struct 中，和当前进程无关
      • 页缓存是全局共享的，所有进程 mmap/read 同一个文件都共享一套 address_space

    🔁 读流程总结

    假设你要读 inode=3 文件的第 1~1024 字节（第 0 页）：

    1. 文件系统查 inode 3 ⇒ 得到 struct inode
    2. 读取页缓存 ⇒ 使用 inode->i_mapping（即 address_space）
    3. 在 i_mapping->i_pages 中查找 index=0 的页
    4. 如果命中，直接读页
    5. 如果 miss：
      • 调用 a_ops->readpage() 从磁盘加载页
      • 将页插入 xarray（key=0）

    address_space 在 Linux 内核中是文件页缓存的核心结构，其用途远不止普通读写场景。下面我们从多个角度系统性地解释 address_space 的其他应用场景，尤其是 mmap 对它的影响。

    🔧 一、address_space 的应用场景

    1. 普通文件读写（read/write）

    这是你最熟悉的场景：

    • 调用 read() 或 write() 时，最终通过 inode->i_mapping 访问页缓存（address_space）。
    • 如果没有命中页缓存，会触发 page fault，由 a_ops->readpage() 填充页缓存。

    2. mmap 内存映射文件

    内存映射文件是 address_space 的重要使用场景之一。

    mmap 时：

    • 创建一个 vm_area_struct（VMA）并关联到 file->f_mapping，即 inode->i_mapping。
    • 每次访问映射区（触发 page fault）时，走 filemap_fault()，通过 address_space->i_pages 查找/加载页。

    作用：

    • mmap 共享文件时，所有进程的 VMA 映射都指向同一个 address_space。
    • 保证多个进程访问相同映射区域时，共享页缓存，避免重复 I/O。

    补充：

    再经由 file->f_mapping → address_space，使用其 page cache。

    3. 页写回（Writeback）与脏页回收

    • 脏页由 mark_page_dirty() 标记，存入 address_space。
    • 回写时调用 a_ops->writepage() 等函数。
    • 所有脏页都由 address_space 管理。

    4. Direct I/O 与 Page Cache 协调

    • 虽然 Direct I/O 绕过页缓存，但文件的元数据（如大小）仍然依赖 inode 和 address_space。
    • 写入前必须确保 address_space 中的脏页被 flush，否则数据不一致。

    5. 文件截断 (truncate)、hole punch（打洞）

    • 截断文件时（如 ftruncate()），需要从 address_space->i_pages 中移除相关页。
    • 释放内存、更新 inode 信息。

    6. 写时复制（COW）机制支持

    • 多个进程 mmap 同一个文件时，address_space 是页共享的桥梁。
    • 写入时触发写时复制，更新 address_space 页表，避免数据冲突。

    📌 address_space 中关键成员参与的作用

    成员 作用
    i_pages (xarray) 管理缓存页，所有 page 都挂在这里
    a_ops 指向 address_space_operations，如 readpage, writepage, invalidatepage
    i_mmap / i_mmap_rwsem 管理 mmap 的所有 VMA（共享和私有）
    writeback_index 指定从哪个页开始写回
    nrpages 当前缓存页数
    invalidate_lock 保证缓存页和磁盘数据一致性

    💬 mmap 对 address_space 的影响

    行为 影响
    mmap 文件 建立 vm_area_struct 并关联到 address_space
    page fault 使用 address_space->a_ops->readpage() 读取数据页
    munmap 更新 i_mmap，释放页引用
    写时复制 使用 address_space 作为原始页的来源

    📎 特殊说明：shmem 和 tmpfs 的 address_space

    不仅磁盘文件使用 address_space，内存文件系统如 tmpfs 也使用它：

    • tmpfs 文件数据页也挂在 i_mapping->i_pages 上。
    • 例如 /dev/shm/、匿名共享内存页。
