    一个简单文件系统实现

    发表于 2025-05-14 更新于 2025-06-05 分类于 linux

    🧩 概要

    ​ 本文介绍了一个非常简单但功能完整的Linux文件系统，名为uxfs。虽然这个文件系统不完善，但它提供了足够的接口和功能，让用户能够创建目录、常规文件，以及对常规文件进行读写操作。它还提供了一个mkfs命令和一个简单的fsdb命令。

    💡设计思路

    ​ 文件系统的整体结构如下图所示，其主要设计要点如下：

    • 文件系统使用固定大小的512字节块。

    • 文件系统块数量是固定的，除去超级块和inode块，数据块总数为470个。

    • inode数量固定为32个。其中inode 0和1保留，inode 2分配给根目录，inode 3分配给lost+found目录，剩下的28个inode可供用户使用。

    • 超级块存储在块号0的位置，占用一个块。超级块中包含两个数组，分别用于记录inode和数据块的使用状态，从而避免了位图操作，使源代码更易于理解。超级块还维护了当前空闲inode和空闲数据块的数量。

    • 每个数据块对应一个inode。第一个inode存储在第 8 块。由于inode 0和1被保留，根目录的inode存储在第10块，lost+found目录的inode存储在第11块。剩余的inodes分布在第12到第39块之间。

    • 第一个数据块从第33块开始。当文件系统被创建时，第50块用于存储根目录的目录项，第51块用于存储lost+found目录的目录项。

    • 每个inode最多只包含9个直接数据块指针，因此最大文件大小被限制为9 × 512字节，共4608个字节。

    • 目录项大小固定，包含一个inode编号和一个28字节的文件名。每个目录项的总大小为32字节。

    设计文件系统，需要确定要支持哪些内核接口。除了读取与写入常规文件、创建与删除目录之外，还需决定是否支持硬链接、符号链接、重命名等功能。

    为此，需要了解文件系统可以导出的各种操作函数集合。主要有以下四种操作结构需要导出：

    • super_operations
    • file_operations
    • address_space_operations
    • inode_operations

    除了决定哪些操作由uxfs自己实现外，还可以选择使用内核提供的通用（generic）函数来代替自定义实现，这在很大程度上简化了文件系统的开发工作。

    🧱 struct buffer_head 字段详解：

    🔹unsigned long b_state

    • 作用：记录缓冲区的状态，是一个 bitmap。
    • 常见状态位（定义于 include/linux/buffer_head.h）：
      • BH_Uptodate：数据是最新的；
      • BH_Dirty：缓冲区已修改，需要写回；
      • BH_Lock：正在进行 I/O 操作；
      • BH_Mapped：已与块设备地址映射；
      • BH_New：新分配的块。

    🔹struct buffer_head *b_this_page

    • 作用：指向同一页中其他 buffer_head 的循环链表。
    • 每个页可能被分成多个 buffer（例如 ext2 下一个页 4096B，分 4 个 1KB buffer），这形成一个 buffer head 的循环链表。

    🔹联合体：union { struct page *b_page; struct folio *b_folio; }

    • 作用：指向这个 buffer 所在的页（或 folio）。
    • Linux 5.16+ 开始逐步引入 folio（一种更高效的 page 管理方式）。
    • 这个 union 是为了兼容老式 page 与新式 folio，在访问时二选一。

    🔹sector_t b_blocknr

    • 作用：这个缓冲块所映射的设备上的起始块号（block number）。
    • 用于定位磁盘上的真实数据位置，单位是块（不是扇区）。

    🔹size_t b_size

    • 作用：表示该缓冲区的大小（单位是字节），比如 1024B、4096B。
    • 注意，不一定等于页大小。通常小于或等于 PAGE_SIZE。

    🔹char *b_data

    • 作用：指向缓冲区在内存中的数据地址，属于页缓存的一部分。
    • 文件系统或块层通过它直接读写数据内容。

    🔹struct block_device *b_bdev

    • 作用：该缓冲块所属的块设备（如 /dev/sda）。
    • 与 b_blocknr 一起定位该缓冲块在设备上的真实位置。

    🔹bh_end_io_t *b_end_io

    • 作用：当 I/O 操作完成时被调用的回调函数。
    • 通常用于解锁 buffer，标记状态（如 BH_Uptodate），唤醒等待线程等。

    🔹void *b_private

    • 作用：供 b_end_io 使用的私有数据指针。
    • 可传递上下文，例如 bio、任务队列或回调状态。

    🔹struct list_head b_assoc_buffers

    • 作用：将该 buffer_head 挂入某个 address_space 的 buffer 列表中。
    • 比如 inode 的 mapping 管理其相关缓冲块，用这个字段组织起来。

    🔹struct address_space *b_assoc_map

    • 作用：该缓冲区所属的 address_space。
    • 通常是 inode->i_mapping，用于建立磁盘数据与文件页缓存之间的关系。

    🔹atomic_t b_count

    • 作用：引用计数，表示有多少用户正在使用该 buffer_head。
    • 用于资源释放控制（当引用为 0 可被释放）。

    🔹spinlock_t b_uptodate_lock

    • 作用：序列化同一页中多个 buffer 的 I/O 完成过程。
    • 通常只用于一个页中的第一个 buffer，防止多个 buffer 同时修改页面状态造成竞态。

    补充：

    1. 用户命名空间（user namespace）

      ​ User Namespace是Linux内核的一项功能，它允许进程拥有与其在宿主机上不同的用户ID (UID) 和组ID (GID) 映射。这意味着一个进程可以在其自己的User Namespace内拥有root权限（UID 0），但在宿主机上却是一个普通用户，从而实现权限的隔离。

      User Namespace是在Linux Kernel 3.8版本中引入的。从Linux 3.8开始，非特权进程也可以创建User Namespace，这为容器技术的发展提供了重要的基础。

      解决了什么问题？

      1. 权限隔离和安全性增强：
        • 在传统的Linux系统中，如果一个进程需要执行某些特权操作（例如，挂载文件系统），它通常需要root权限。然而，root权限是全局的，如果一个拥有 root 权限的进程被攻破，攻击者就可以控制整个宿主机系统。
        • User Namespace允许进程在命名空间内部被认为是root用户，但实际上在宿主机上映射到一个非特权用户。这样，即使容器内的root用户被攻破，其权限也仅限于该User Namespace内部，无法对宿主机系统造成危害，大大增强了安全性。
        • 这对于构建“无根容器”（rootless containers）至关重要，它允许普通用户运行容器，而无需root权限或依赖setuid程序，降低了安全风险。
      2. 多用户环境下的灵活权限管理：
        • 在容器化环境中，你可能希望容器内的应用程序以特定的用户身份运行，而不是宿主机上的真实用户。User Namespace允许你灵活地将容器内部的UID/GID映射到宿主机上的不同UID/GID范围，从而实现更精细的权限控制。
      3. 简化特权操作：
        • 一些应用程序在容器内需要以root身份运行才能正常工作（例如，安装软件包）。User Namespace使得这些应用程序可以在隔离的环境中获得必要的特权，而不会影响宿主机的安全性。
