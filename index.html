参数包

发表于 2025-07-04 分类于 C++

在C++中，pack（参数包）是一种可接受多个元素的语言实体，包含以下几类：

  • 参数包（parameter pack）

    • 模板参数包（template parameter pack）
    • 函数参数包（function parameter pack）

  • Lambda初始化捕获包（lambda init-capture pack）(C++20起)

  • 结构化绑定包（structured binding pack）(C++26起)

模板参数包是一种模板参数，它接受零个或多个模板实参（可以是常量、类型或模板）。函数参数包是一种函数参数，它接受零个或多个函数实参。

Lambda初始化捕获包是一种Lambda捕获，它为其初始化器中包展开的每个元素引入一个初始化捕获。(C++20起)

结构化绑定包是结构化绑定声明中的一个标识符，它引入零个或多个结构化绑定。(C++26起)

一个包的元素数量等于：

  • 如果包是模板或函数参数包，则等于为其提供的实参数量。
  • 如果包是Lambda初始化捕获包，则等于其初始化器中包展开的元素数量。(C++20起)
  • 如果包是结构化绑定包，则等于初始化器的结构化绑定大小减去结构化绑定声明中非包元素的数量。(C++26起)

至少包含一个参数包的模板被称为可变参数模板（variadic template）。

语法

模板参数包（出现在别名模板、类模板、变量模板（C++14起）、概念（C++20起）和函数模板的参数列表中）：

  1. type ... pack-name(可选)：一个带有可选名称的常量模板参数包。
  2. typename|class ... pack-name(可选)：一个带有可选名称的类型模板参数包。
  3. type-constraint ... pack-name(可选)：一个带有可选名称的受限类型模板参数包。(C++20起)
  4. template < parameter-list > class ... pack-name(可选) (C++17前)或template < parameter-list > typename|class ... pack-name(可选)(C++17起)：一个带有可选名称的模板模板参数包。

函数参数包（一种声明符形式，出现在可变参数函数模板的函数参数列表中）：

  1. pack-name ... pack-param-name(可选)：一个带有可选名称的函数参数包。

对于非参数包的语法，请参见Lambda初始化捕获包和结构化绑定包(C++26 起)。(C++20起)

mount的时候将fd和挂载点的信息传递给内核。

✅ MS_RDONLY

  • 含义：将文件系统挂载为只读。
  • 影响：你不能对该文件系统写入文件或更改元数据。
  • 常用于：挂载 CD-ROM、ISO 镜像、或者只读模式修复文件系统。

✅ MS_NOSUID

  • 含义：挂载后该文件系统内的所有文件即使有 setuid（SUID）或 setgid（SGID）位，也会被忽略。
  • 影响：
    • 即使程序文件设置了 SUID（比如 /usr/bin/passwd），执行时也不会提升权限。
  • 用途：
    • 挂载 USB、SD 卡、网络文件系统时，防止恶意程序借助 SUID/SGID 提权。

✅ MS_NODEV

  • 含义：挂载后该文件系统中的设备 special 文件（即字符设备文件和块设备文件）无法使用。
  • 影响：即使你创建了 /dev/sda 这样的设备文件，程序也不能通过它访问设备。
  • 用途：
    • 安全限制：挂载某些不可信来源的文件系统（如 U 盘、loop 文件），防止通过 fake device file 做非法操作。

在 FUSE（Filesystem in Userspace）中，struct mount_opts 用于传递挂载参数，控制文件系统的行为。以下是各字段的详细说明：

字段含义及作用

1. 权限控制

字段 类型 说明
allow_other int 是否允许非挂载用户访问文件系统（默认仅挂载用户可访问）。 需配合 /etc/fuse.conf 中的 user_allow_other 选项使用。
allow_root int 是否允许 root 用户访问文件系统（默认允许）。 若禁用，root 用户将无法访问。

2. 挂载行为控制

字段 类型 说明
nonempty int 是否允许挂载到非空目录（默认禁止）。 启用后，原有目录内容会被隐藏，直到卸载。
blkdev int 是否挂载的是块设备（而非普通文件系统）。 用于特殊场景（如直接挂载磁盘镜像）。

3. 辅助标志

字段 类型 说明
ishelp int 是否显示帮助信息（由 -h/--help 触发）。
flags int 挂载的全局标志位，例如： - MS_RDONLY（只读挂载） - MS_NOSUID（忽略 SUID 权限）。

4. 标识与分类

字段 类型 说明
fsname char* 文件系统的名称标识（通过 -o fsname=NAME 指定）。 用于 df 或 /proc/mounts 中显示。
subtype char* 文件系统的子类型（如 fuse.sshfs）。 影响 /proc/mounts 中的显示格式（type.subtype）。
subtype_opt char* 子类型的挂载选项（内部使用）。

5. 选项传递

字段 类型 说明
mtab_opts char* 写入 /etc/mtab 的选项字符串。
fusermount_opts char* 传递给 fusermount 工具的选项。
kernel_opts char* 传递给内核的原始挂载选项（未解析的 -o 参数）。

发表于 2025-07-01 更新于 2025-07-03 分类于 Linux

概述

FUSE是File system in User Space，它是一个用户空间的文件系统框架，通过fuse，程序员可以在用户态实现文件系统，并且不需要特权用户的支持。

fuse包括三个模块：

  • 内核模块：实现了一个完整文件系统的框架，但具体操作没有实现（由程序员在用户空间实现）。
  • 用户空间库：提供编程接口，程序员通过实现fuse提供的fuse_operations或fuse_ll_operations的所有或部分操作来实现文件系统
  • mount工具：fusermount用于挂载用fuse实现的文件系统。

fuse源码中提供了几个简单的例子，在src/example目录下，示范如何用fuse开发用户空间的文件系统。

  • null.c,hello.c：一个简单的使用fuse的例子，演示了使用fuse建立本地用户空间文件系统的基本过程

  • hello_ll.c：与上述hello.c实例实现的功能一样，只不过在具体实现代码中调用了更加低层的fuse函数来完成。

  • fu***mp.c：这个实例实现了用fuse完全挂载本地文件系统挂载到用户空间文件系统的过程，展示了比较完整的fuse的使用过程。

  • fu***mp_fh.c：这个实例是上面./fu***mp的扩展（使得基于fuse的这个文件系统更加的完善，例如：增加了fflush、lock等的支持）。

学习理解fuse源码

l kernel/inode.c —> 主要完成fuse文件驱动模块的注册，提供对supper block的维护函数以及其它(驱动的组织开始文件)

l kernel/dev.c —> fuse 的(虚拟)设备驱动

l kernel/control.c —> 提供对于dentry的维护及其它

l kernel/dir.c —> 主要提供对于目录inode索引节点的维护

l kernel/file.c —> 主要提供对于文件inode索引节点的维护

l lib/helper.c —> “fuse_main()”调用的主入口

l lib/fuse_mt.c —> fuse 的mount管理

l lib/fuse.c —> lib库主框架文件，实现了主要框架及对”用户实现的文件系统操作代码”的封装

l lib/fuse_lowlevel.c –> 实现比较底层的函数封装，供fuse.c等使用

l lib/fuse_loop.c —> fuse lib循环监视”fuse driver”的通信缓存

l lib/fuse_loop_mt.c —> 同上

l lib/fuse_session.c —> fuse会话管理

FUSE挂载参数

fuse提供一系列挂载选项，如direct_io(跳过页缓存），allow_root(允许root访问挂载的文件系统），allow_other（允许其他用户访问挂载的文件系统），nonempty（允许把文件系统挂载到非空目录），big_writes（支持大于4k的写操作，仅fuse2.8以上版本支持）等，更详细的参数说明参见源码包中的文档。

使用fuse会给文件系统带来性能上的损失，因涉及到多次内核态到用户态的切换，我通过测试本地ext3和fuse-ext3的读写性能进行对比分析发现：

l 在cache情形下，文件系统的读写请求都通过页高速缓存满足，fuse-ext3的性能只有ext3的15%~30%；

l 在sync-write情形下，读与上相同，写的性能两者差不多，因设计到磁盘操作，fuse的开销就显得微不足道。

BPE Tokenizer Training

BPE分词器的训练主要包含以下三个步骤：

词表初始化

分词器的词表将一个字节串一对一映射到到整数。由于我们训练的是字节级分词器，单字节表示的范围是$[0-255]$，所以初始词表的大小为 256。

预分词（Pre-tokenization）

有了初始词表后，理论上你可以统计文本中相邻字节对出现的频率，并从出现最频繁的字节对开始进行合并。然而，这种方法计算成本很高，因为每合并一次都需要对整个语料库完整地遍历一遍。

除此之外，如果直接在整个语料库上合并字节，还可能导致一些仅在标点上不同的词被赋予完全不同的token ID，例如 dog!和dog.。尽管它们在语义上非常接近（仅标点不同），它们却会被视为完全不同的标记。

为了解决这个问题，我们引入了预分词。预分词可以看作是对语料库的一种粗粒度的分词方法，它帮助我们统计字符对出现的频率。假设单词text出现在语料中10次，我们就可以将字符t和e紧邻出现的次数加10，而不必在语料中一遍一遍地查找。因为我们训练的是字节级BPE模型，每个预分词的标记都是一串UTF-8编码的字节。

Sennrich等人在其原始BPE实现中，采用了简单的空格作为预分词策略（即s.split(" ")）。我们采用GPT-2所用的基于正则表达式的预分词器（Radford等人，2019），该正则表达式来源于openai/tiktoken 的 GitHub PR #234：

这个正则表达式可以识别常见缩写、字母、数字、标点符号和空白字符，从而实现更合理的预分词效果。

在实际使用该方法时，建议使用re.finditer而非re.findall，以避免在构建“预分词（pre-token）”到词频的映射过程中占用过多内存。

计算BPE合并对

在将输入文本转换为预分词，并将每个预分词表示为UTF-8字节序列后，我们便可以开始计算BPE合并对。从整体上看，BPE 算法是一个迭代过程：每轮都会统计所有相邻字节对的频率，并找出出现次数最多的一对（例如A和B）。随后，将文本中所有该频率最高的字节对(A, B)合并成一个新的token（如AB）。这个新token被添加到词表中。

最终词表的大小 = 初始词表大小（本例中为256）+ 训练过程中执行的BPE合并操作次数。

⚠️ 为了提高训练效率，BPE不会考虑跨越预分词边界的字节对。

Sennrich等人在2016年用于机器翻译的原始BPE实现中，会在每个词的末尾加一个特殊符号，比如</w>，用来表示词的结束。这样做有两个目的：

  • 帮助模型区分像low和lower（它们共享前缀，但是不同词）
  • 避免跨词错误合并（比如"the" + "m"不该变成"them"）

当我们使用Byte-Level BPE时，需要再手动加</w>之类的词边界标记，原因是：

  • 所有字节都被编码，包括空格、标点、换行符等
  • 空格' '本身就是一个字节，会参与BPE合并和训练

当遇到多个频率相同的候选字节对时，应按字典序优先选择较大的那一对。例如，若("A", "B")、("A", "C")、("B", "ZZ")和("BA", "A")都具有最高频率，则选择 ("BA", "A")：

特殊Token的处理

在训练和编码过程中，我们通常需要保留一些特殊字符串（如<|endoftext|>）以编码元信息（例如：文档边界）。这些特殊token不应被进一步拆分，必须始终作为一个整体的token保留，并赋予固定的token ID。

假设<|endoftext|>代表文本结束，模型在遇到它时应停止生成，因此它必须作为一个独立的token保留在词表中，确保编码成唯一的整数ID。

kobject对象

发表于 2025-06-24 分类于 Linux

📝概述

在Linux设备模型中，kobject是一个核心概念，它提供了一种统一的方式来管理和组织内核对象。你可以把它想象成一个基石，许多更复杂的内核结构（如kset、device、driver等）都是建立在它之上的。

🎯kobject是什么？

从根本上说，一个kobject是一个嵌入在其他内核数据结构中的结构体，主要作用是：

  • 引用计数：这是kobject最重要的功能之一。每个kobject有一个引用计数。当有代码需要使用一个内核对象时，引用计数会增加；当不再需要时，引用计数会减少。当引用计数降到零时，kobject会被自动销毁。这有效防止了对象被过早释放，导致“悬空指针”等问题。
  • 管理父子关系：kobject可以形成树状结构，每个kobject可以有一个父kobject。这在设备模型中尤为重要，因为它能反映设备在硬件拓扑中的层次关系（例如，一个USB设备插在一个USB控制器上）。
  • sysfs表示：每个kobject在/sys文件系统中都有一个对应的目录。这使得用户空间程序能够通过文件系统接口查看和操作内核对象的信息，实现了内核和用户空间的交互。

🐛kobject的用途

kobject自身不承载太多具体的业务逻辑，它更像是一个“基类”或“框架”。它通常被嵌入到更高级的内核结构中，例如：

  • struct device：代表一个物理或虚拟设备。device结构体内部包含一个kobject。
  • struct device_driver：代表一个设备驱动。device_driver结构体内部也包含一个kobject。
  • struct kset：用于组织和管理一组相关的kobject。kset自身也包含一个 kobject。

通过kobject提供的通用功能（引用计数、sysfs表示、父子关系），这些复杂的内核结构可以共享一套统一的管理机制。

🧬总结

kobject是Linux设备模型中一个非常基础但至关重要的构建块。它提供了统一的对象管理机制，通过引用计数实现了对象的生命周期管理，并通过在sysfs中创建文件系统接口方便了用户空间对内核对象的访问。理解kobject的工作原理，是深入理解Linux设备模型、驱动开发以及内核内部机制的关键一步。
