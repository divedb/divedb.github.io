UNQUALIFIED NAMES

  • Unqualified names undergo a two step process: unqualified lookup and “maybe” ADL
  • Let U be the set produced by (U)nqualified lookup and A the set produced by (A)DL
  • If the set U contains any of the following:
    • A declaration of a class member
    • A block scope function declaration that is not a using declaration
    • A declaration that is neither a function nor a function template

Then A is empty and the lookup does not proceed to step 2.

STEP 2: ADL

For each argument type T in a function call there are two sets (each of size zero or more)

  • Set E (assoicated entities)
  • Set N (associated namespaces)

typedefs and using-declarations do not! contribute to this set.

  • T is a fundamental type.

  • T is of class type.

    • E is [class itself, the class which is a member to (if any) and any indirect or direct base classes]

    • N is [innermost enclosing namespace of its associated entities]

      • Step 1. Unqualified lookup finds nothing U is empty.
      • Step 2. E(pipe, user_pipe) and N[lib, user]. ADL finds [lib::connect, user::connect]
      • Overload resolution picks user::connect() as a better match
  1. T is an enum
    • E is [the enum itself, if it’s a class member: the member’s class]
    • N is [innermost enclosing namespaces of its declaration]

  1. T is a pointer to X or an array of X.

    N,E is [entities associated with X]

    • Unqualified lookup finds nothing. U is empty.
    • ADL finds, E(lib::pipe), N[lib]. Therefore, we feed. [lib::connect(pipe* ptr)] to the overload resolution

  2. T is a function type

    N, E is [function parameter types, return type]

    • Unqualified lookup finds nothing. U is empty.
    • ADL finds, E[lib::pipe], N[lib]. Therefore, A is [lib::process()]

    N, E is [rule#5, rule#2]

  1. T is a pointer to data member of a class X.

N, E is [rule #2, associated namespaces and entities of the member type]

HACK #29 数组非法访问导致内存破坏

1. 数组错误操作

错误的操作数组导致的典型bug之一就是缓冲区溢出，例如向已分配的内存空间之外写入数据。如果是这类bug发生在栈上的缓冲区中，就可能引发安全漏洞。因此出现了许多预防和应对措施，例如通过指定缓冲区大小来编写安全的函数、源代码检查工具、编译器在构建时的报警等。即便如此，这种bug仍时有发生。此外，在计算数组的下标时，如果计算方法有误，就可能算出负的下标，也可能会引起缓冲区溢出（早期PostgresSQL代码中可以看到arr[-1]用法，关键还是要看对应的内存是否有分配）。本hack后半部分也会讲述这种bug的调试方法。

2. 可以怀疑是缓冲区溢出的情况

栈帧的#0、#1显示的地址上很难放置程序、共享内存等。大多数i386架构的Linux发行版中，程序被定位到0x08000000地址附近，共享库北定位到0xb0000000之后的地址。在x86_64中，程序定位到0x400000或0x0600000，共享库被定位到0x3000000000、0x2aaaaaaa0000地址附近，据日情况由链接器决定。不同的使用环境下，与链接器的定位地址有关的选项也可能发生变化，因此记住在调试对象的环境中，标准情况下程序和共享库会被定位到什么地址，在阅读backtrace时就会很方便。

3. 运行地址的改变

改变程序运行地址的方法基本可以分为3类：

  1. 指定地址并调用。C语言中的if或for语句等进行条件判断时就会用到这种方法，调用同一源代码内的函数时也会使用这种方法。

  2. 指定了一块内存区域，其中保存了跳转地址。

  3. 执行ret指令，用于函数结束时返回调用者函数。ret命令将栈指针指向的位置的值作为跳转（返回）地址使用。

如果bug破坏了这些方法用到的值（被错误的地址覆盖），就可能跳转到错误的地址。但是，第1类方法使用的地址一般保存在只读空间内。所以，尝试破坏该地址就会产生segmentation fault。此时，core文件中会记录这一瞬间的程序计数器的值，因此分析也比较容易。

相反，第2、第3类方法使用的地址位于GOT（Global Offset Table）或栈等可写的空间中，因此即使bug破坏其内容，也无法立即被检测到。

4. 确定破坏跳转地址值的位置（栈破坏）

简单来说，确定破坏地址值的位置的方法，就是将错误地址当做数据，寻找复制该数据的地方。以本hack开头的栈跟踪为例，我们要寻找错误的写入0x20656c62这个数据的地方。

很重要的是需要怀疑数据是否为字符串的一部分，因为错误的将数据写入地址的典型情况之一就是字符串复制。本hack例子中地址0x20656c62可以认为是字符串” elb”。

Interfaces

概要

接口是程序中两个部分之间的约定。明确提供者和使用者的期望是十分重要的。拥有良好的接口（易于理解、高效使用、减少出错、支持测试等）可能是代码组织中最重要的一方面。

  • I.1: Make interfaces explicit
  • I.2: Avoid non-const global variables
  • I.3: Avoid singletons
  • I.4: Make interfaces precisely and strongly typed
  • I.5: State preconditions (if any)
  • I.6: Prefer Expects() for expressing preconditions
  • I.7: State postconditions
  • I.8: Prefer Ensures() for expressing postconditions
  • I.9: If an interface is a template, document its parameters using concepts
  • I.10: Use exceptions to signal a failure to perform a required task
  • I.11: Never transfer ownership by a raw pointer (T*) or reference (T&)
  • I.12: Declare a pointer that must not be null as not_null
  • I.13: Do not pass an array as a single pointer
  • I.22: Avoid complex initialization of global objects
  • I.23: Keep the number of function arguments low
  • I.24: Avoid adjacent parameters that can be invoked by the same arguments in either order with different meaning
  • I.25: Prefer empty abstract classes as interfaces to class hierarchies
  • I.26: If you want a cross-compiler ABI, use a C-style subset
  • I.27: For stable library ABI, consider the Pimpl idiom
  • I.30: Encapsulate rule violations

module_init和module_exit

1 概要

​ 在Linux内核模块编程中，module_init和module_exit是两个至关重要的函数，它们分别负责模块的初始化和卸载。理解这两个函数对于编写和管理内核模块至关重要。

1.1 module_init
  • 作用： module_init函数是内核模块的入口点。当模块被加载到内核时，这个函数会被调用。它负责执行模块的初始化工作，例如：
    • 注册设备驱动程序
    • 分配内存
    • 初始化数据结构
    • 创建sysfs文件系统
    • 等等

1.2 module_exit
  • 作用： module_exit函数是内核模块的出口点。当模块从内核卸载时，这个函数会被调用。它负责执行模块的清理工作，例如：
    • 注销设备驱动程序
    • 释放内存
    • 销毁数据结构

1.3 注意事项
  • module_init 和module_exit函数必须成对出现，一个模块必须同时定义这两个函数。

__initdata 是 Linux 内核中用于标记初始化数据区域的一个特定符号。它的作用是在内核启动过程中标记那些只会在内核初始化阶段使用的数据，帮助优化内存使用，并且允许操作系统在初始化后释放这些数据区域，以便为其他任务腾出空间。

1. __initdata 的用途

在内核中，很多数据结构只在启动时使用。例如，一些与初始化过程相关的静态数据，在内核初始化完成后就不再需要。为了节省内存，可以将这些数据标记为 __initdata，并在内核加载时将其放置在初始化区域中，初始化完成后这些区域可以被释放或回收。

  • 优化内存：通过将这些数据标记为 __initdata，编译器可以将它们放置在特定的内存区域，从而在系统初始化完成后能够清理这些数据区域，释放内存。
  • 仅初始化阶段使用：标记为 __initdata 的数据仅在初始化阶段有效，之后会被丢弃，减少了内存占用。

2. __initdata 的定义

__initdata 是由内核构建系统（通过链接脚本）定义的一个符号，表示某些数据仅在初始化时使用。这通常与 __init 修饰符一同使用。

  • __initdata 是一个符号，它的作用是将数据放置到内核的特殊段中。
  • __init 修饰符通常用于函数，标记这些函数仅在初始化时执行，而 __initdata 是用于数据的标记。

4. __initdata 与 __init 的区别

  • __init 修饰符是用于函数，标记该函数仅在初始化阶段执行，初始化完成后会被丢弃。
  • __initdata 修饰符是用于数据，标记该数据仅在初始化阶段有效，初始化后将被释放或清理。

5. 内存管理与优化

在 Linux 内核中，使用 __initdata 和 __init 的一大好处是可以减少初始化后不再需要的数据占用内存。内核在启动时会将这些标记的数据和代码放到内存的专用初始化区域中，在初始化结束后，这些区域会被回收。

6. 与 __exitdata 的对比

与 __initdata 相对的是 __exitdata，它用于标记退出时才需要的数据。虽然 __exitdata 不如 __initdata 常见，但它也可以用于优化一些在模块卸载时才需要的数据。

总结

  • __initdata 是内核中用于标记初始化时使用的数据区域。
  • 它帮助内核优化内存，释放那些在初始化后不再使用的数据。
  • 在编写内核代码时，只有在数据仅在初始化阶段需要时，才应使用 __initdata 来标记这些数据，以便内核能够释放不必要的内存。

__init 是与内核构建系统紧密相关的一个修饰符，用于标记内核中仅在初始化阶段使用的函数。它并不是 GCC 或动态链接库本身的直接处理，而是由内核的构建工具链和链接器来处理。

具体来说，__init 的作用和工作原理如下：

1. __init 的工作原理

  • GCC 处理：__init 是一个由 GCC 处理的修饰符。它用于指示编译器将标记为 __init 的函数和代码放置到内核的一个特殊部分——初始化段（.init.text）。这些函数在内核初始化期间执行，且仅在初始化期间有效。

  • 链接器处理：当内核源码编译完成后，链接器（如 ld）根据内核的链接脚本将这些被标记为 __init 的函数和数据放置到专用的初始化段中。内核启动时，这些段中的内容会被加载并执行。

    在初始化完成后，内核会丢弃这些初始化段，从而节省内存空间。这是通过链接器的垃圾回收机制（如 discard）实现的。内核链接器会丢弃 .init.text（存放初始化函数）和 .init.data（存放初始化数据）段中的内容。

2. __init 在 GCC 中的处理

GCC 会将使用 __init 标记的函数放置到一个特殊的段中，例如 .init.text。这是由 GCC 编译器和内核的构建脚本配合实现的。以下是一个简化的流程：

  • 定义 __init 函数：

  • 编译时： GCC 会将 __init 标记的函数放入内核的 .init.text 段中。该段是内核启动时的一部分，执行完初始化后会被丢弃。

  • 链接时： 链接器会根据内核的链接脚本，将 .init.text 段中的函数放入内核映像中，等初始化完成后，这些段中的内容会被标记为不可访问，释放掉这部分内存。

3. __init 与 动态链接库

__init 并不是动态链接库（动态库）处理的一部分。动态链接库的功能通常与应用程序的运行时连接相关，而 __init 是内核编译和链接阶段的一部分，用来优化内核内存管理。

在内核中，__init 修饰符标记的函数或数据，只会在内核初始化时有效，在初始化完成后，这部分代码会被丢弃。这与动态链接库的加载和卸载机制是不同的，因为动态链接库的内容在运行时动态加载，并且通常在程序的生命周期内保持有效。

4. GCC 的实现

  • GCC 的 __init 标记：在编译阶段，GCC 会根据 __init 修饰符为函数或数据分配一个特殊的内存段，通常是 .init.text（用于代码）或 .init.data（用于数据）。

    这些段在内核初始化后会被释放或丢弃，确保内核在启动完成后不再占用不必要的内存。

  • 内核的链接脚本：链接脚本会将这些初始化段（.init.text 和 .init.data）标记为“初始化时有效”，并在内核初始化完成后释放这部分内存。

5. 总结

  • __init 是由 GCC 处理的修饰符，用于将函数和数据放入内核的初始化段中。
  • 这些初始化段（.init.text 和 .init.data）在内核初始化完成后会被丢弃，释放内存。
  • 它与动态链接库无关，动态链接库的加载和卸载是在运行时处理的，而 __init 的工作发生在内核的编译和链接阶段。

通过使用 __init 修饰符，内核能够有效地管理内存，减少初始化后不再需要的代码和数据的占用。

HACK#9 调试时必需的栈知识

1. 概要

栈（stack）是程序存放数据的内存区域之一，其特征是LIFO（Last In First Out，后进先出）式数据结构，即后放进的数据最先被取出。向栈中存储数据的操作称为PUSH（压入），从栈中取出数据称为POP（弹出）。在保存动态分配的自动变量时要使用栈。此外在函数调用时，栈还用于传递函数参数，以及用于保存返回地址和返回值。

栈上依次保存了传给函数的参数、调用者的返回地址、上层栈帧指针和函数内部使用的自动变量。此外，调用函数时还会用栈来临时保存寄存器。每个函数都独自拥有这些信息，称为栈帧（stack frame）。此时需要适当地设置表示栈帧起始地址的帧指针（FP）。此外，栈指针（SP）永远指向栈的顶端。

x86_64中自动变量和工作空间有可能会超过栈指针。从栈指针指向的地址开始，再次扩展128字节，这部分空间称为red zone（“危险区”），可以当作自动变量和工作空间使用。这是AMD64的ABI规格定义的内容。
