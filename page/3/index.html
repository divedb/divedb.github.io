btrfs实验

发表于 2025-06-02 更新于 2025-06-03 分类于 linux

📘 btrfs设计

​ btrfs的实现依赖于简单且广为人知的结构。它在性能方面表现良好，但其长期目标是在文件系统不断增长和老化的过程中保持性能稳定，这比短期跑分表现更为重要。为此，btrfs使用基准测试来模拟整个文件系统生命周期中的性能表现。

📗 B树数据结构

​ btrfs的B树是一个通用的数据存储机制，能够存储多种不同类型的数据。在内部，它仅识别三种结构：

  • 键（key）
  • 项（item）
  • 块头（block header）

树的上层节点只包含[key, 块指针]对。而树的叶子节点被划分为两个相对生长的区域：一侧是固定大小的item项数组，另一侧是可变大小的item数据区。每个item的offset和size字段指定了该项数据在叶子节点中的存储位置。

每个item的数据大小不固定，btrfs使用不同类型的item数据来定义各种文件系统结构。btrfs_disk_key中的type字段指明了该项存储的数据类型。

块头结构中包含了如下内容：

  • 当前块的内容校验和
  • 所属文件系统的UUID
  • 当前块在树中的层级
  • 块号（即该块应被存储的位置）

这些字段可在读取数据时对元数据进行完整性验证。所有指向btree块的引用也会一并记录其预期的generation字段，这有助于btrfs检测出媒介上的幽灵写入或误写入的情况。

为了简化文件系统的回写代码，上层节点指针中不会存储下层节点的校验和。节点的generation字段在其被插入B树时就已确定，而校验和只有在实际写盘前才会计算。因此，只用generation字段就可以让btrfs检测到幽灵写入，而无需在每次更新校验和时都回溯并修改上层节点。

generation字段对应的是分配该块时的事务ID，这不仅支持增量备份，也被写时复制（copy-on-write）事务子系统所使用。

📚 文件系统数据结构

​ 文件系统中的每个对象都有一个objectid，在创建时动态分配。一个尚未使用的objectid表现为文件系统B-tree键空间中的一个“空洞”——即该objectid在树中尚不存在。objectid占据键中最高有效位，使得属于同一个对象的所有项目都可以在B-tree中逻辑上聚集在一起。

键中的offset字段表示对象中某项数据的字节偏移量。对于文件数据块（extents），该偏移量表示该数据块在文件中的起始位置。type字段表示该项的类型信息，并预留了扩展空间以备将来使用。

🍃 inode节点

​ inode节点存储在struct btrfs_inode_item中，其键的 offset 固定为 0，type 类型值为 1。对于某个对象，inode 项在 B-tree 中始终具有最低的键值。它存储传统的 stat 数据（如文件大小、权限、时间戳等），但不包含嵌入式文件内容或扩展属性数据，这些数据保存在其他类型的项中。

文件

小文件（尺寸小于一个 B-tree 叶子节点块）可以直接打包存储在 extent 项中。此时，键的 offset 表示文件中该数据的字节偏移，struct btrfs_item 的 size 字段记录实际存储的数据大小。一个文件可以包含多个这样的项目。

大文件的数据则以 extent 形式存储。struct btrfs_file_extent_item 记录该数据块的版本号（generation）及其对应的磁盘位置（磁盘块起始地址及块数）。Extent 项还记录了该数据块在文件中的逻辑偏移及其使用的磁盘块数量。这种方式允许 Btrfs 在无需读取旧数据的情况下直接覆盖某段数据。例如，在一个 128MB 的 extent 中间写入 1MB 数据，可能会生成如下三个 extent 项：

  • [旧 extent: 0-64MB]
  • [新 extent: 1MB]
  • [旧 extent: 65MB–128MB]

文件数据的校验和（checksum）存储在专用的 B-tree 中，结构为 struct btrfs_csum_item。校验和键的 offset 对应于 extent 的起始字节编号。校验在数据压缩或加密之后进行，反映实际写入磁盘的字节。

一个校验项可能包含多个校验和。注意，内嵌于 B-tree 中的小文件数据由所在 B-tree block 的校验和覆盖。

引用计数的 Extent

引用计数是快照子系统的基础。每个分配给 B-tree 或文件的数据块（extent）都会在 struct btrfs_extent_item 中记录引用数量。保存这些项的树结构也充当了磁盘使用情况的分配图。

部分树（如 extent tree 和树根目录）不会对 extent 进行引用计数，仅依赖写时复制日志保护。不过，磁盘上所有分配块都使用相同类型的 extent item。

Extent 块组

Extent block group 将磁盘划分为至少 256MB 的块区，便于优化分配策略。每个 block group 记录可用块的数量。文件和目录优先尝试其所属的首选块组。

Block group 带有标志位，表示其适用于数据或元数据分配。在格式化时，磁盘会被划分为交错的元数据组（约占 33%）和数据组（约占 66%）。尽量避免数据和元数据混存，提高 fsck 检查效率并减少挂载期间写回的寻道开销（但会略微增加读取时的寻道次数）。

Extent 树与设备映射（DM）集成

Btrfs 的 extent 树用于灵活划分磁盘空间，可为某个或一组树根、目录或 inode 提供特定分配策略。管理员可控制文件系统的哪些部分使用条带、镜像或绑定到指定设备。

Btrfs 计划通过与设备映射器（DM）集成，简化大规模存储池管理。原则上每个磁盘可拥有一个 extent 树，管理员可将其分配给子卷、目录或文件。

显式反向引用（Back References）

反向引用有三个主要目的：

  1. 精确追踪引用者：防止错误释放仍被引用的 extent。
  2. 定位引用者：当发现某个块损坏时，能快速找出使用它的对象。
  3. 支持迁移与维护：例如文件系统收缩或存储池维护时迁移块。

文件 extent 的反向引用

一个文件 extent 可能被以下引用：

  • 多个快照、子卷，或同一子卷的不同版本
  • 同一子卷中不同文件
  • 同一文件的不同偏移

  • 子卷根的 objectid
  • 引用树的 generation
  • 引用该 extent 的文件的 objectid
  • 文件中引用的 offset

反向引用键的构造

反向引用包含四个 64 位字段，会被哈希成一个 64 位值作为 key 的 offset。key 的 objectid 为 extent 的起始字节，key 类型为 BTRFS_EXTENT_REF_KEY。

若哈希冲突（offset 溢出），会通过增加 hash 值继续向前查找，直到匹配项出现。

快照与子卷

子卷（subvolume）本质上是一个命名的 B-tree，保存文件与目录。它们有独立 inode，可设置属主与属组。子卷可以设置写入配额，一旦超限禁止写入。子卷中所有块与 extent 都使用引用计数以支持快照功能。最多可创建 2⁶⁴ 个子卷。

快照与子卷结构完全相同，只是在创建时其根块（root block）初始共享另一子卷的根。创建快照时，会增加原根块的引用计数。写时复制机制确保原子性修改，互不干扰。
