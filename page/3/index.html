Linux 内核 dcache（Directory Cache）笔记

一、dcache 简介

dcache 是 Linux VFS（虚拟文件系统）中用于加速路径名解析的核心机制之一。其全称为 directory cache，主要缓存的是路径中每一级目录和文件名对应的 dentry（directory entry）对象。

  • 避免频繁查找磁盘，提高访问效率。
  • 由内核动态维护，可复用、延迟回收。

二、dentry 结构体

三、dcache 的设计要点

  1. 高速路径名查找
    • 采用 hash 表加速路径名查找（d_lookup）。
    • 避免每次都访问磁盘。
  2. 内存中层级结构
    • dentry 构成一棵树（有父子关系）。
    • 每一级目录缓存为一个 dentry。
  3. 引用计数与生命周期
    • 使用 dget()/dput() 控制引用计数。
    • 引用计数为 0 且满足回收条件的 dentry 会被释放或移入 LRU。

四、常用操作函数

1. d_lookup(parent, &name)

  • 在父目录的 dentry 中查找子文件名是否存在于 dcache。
  • 仅查缓存，不查磁盘。

2. d_alloc(parent, &name)

  • 创建一个新的 dentry。
  • 通常结合 d_add() 或 d_splice_alias() 进行加入。

3. d_add(dentry, inode)

  • 将 dentry 与 inode 绑定，并加入父目录中。

4. dput(dentry)

  • 减少 dentry 的引用计数。
  • 若计数为 0，进入回收流程。

五、路径解析过程

以 /foo/bar 为例：

  1. 内核从根开始查找 foo。
  2. 如果 foo 在 / 的 dentry 的子哈希表中，直接命中。
  3. 否则触发文件系统的 lookup 操作从磁盘加载，并创建新 dentry。
  4. 类似地解析 bar。

注意：这个过程完全由 dcache 管理，减少重复磁盘 I/O。

六、内存回收机制

  • 内核会定期回收不再使用的 dentry。
  • shrink_dcache_parent()、shrink_dcache_sb() 等负责批量清理。
  • LRU（最近最少使用）机制与引用计数配合判断释放时机。

特性 描述
提高路径访问速度 避免重复磁盘访问
树形结构 支持完整路径查找

缓存失效机制 支持自动回收

bdev头文件

发表于 2025-06-10 更新于 2025-06-11 分类于 Linux

block/bdev 模块是 Linux 内核中负责管理块设备(block devices)的核心子系统，它提供了块设备抽象层，将各种存储设备(如硬盘、SSD、USB存储等)统一表示为块设备，并处理它们的I/O操作。

核心概念

1. 块设备(Block Device)

块设备是以固定大小块(通常为512字节或4KB)为单位进行随机访问的存储设备，与字符设备(按字节流访问)形成对比。

2. 主要数据结构

struct block_device

表示一个块设备的内核对象，包含：

  • 设备号(dev_t)
  • 设备大小和块大小
  • 指向底层设备的指针
  • 设备状态标志
  • 关联的请求队列

struct gendisk

表示一个通用磁盘(generic disk)：

  • 磁盘名称和主/次设备号
  • 分区表信息
  • 块设备操作集合
  • 容量和扇区大小

struct bio

表示一个块I/O请求：

  • 目标设备
  • 读写方向
  • 内存页/缓冲区
  • 完成回调函数

主要功能模块

1. 块设备注册与管理

  • register_blkdev() - 注册块设备
  • blkdev_get()/blkdev_put() - 打开/关闭块设备
  • bdget() - 通过设备号获取block_device结构

2. 请求队列管理

  • blk_init_queue() - 初始化请求队列
  • blk_cleanup_queue() - 清理请求队列
  • blk_queue_make_request() - 设置自定义请求处理函数

3. I/O调度

  • 多种调度算法(elevator)实现：
    • noop - 简单FIFO
    • deadline - 截止时间调度
    • cfq - 完全公平队列
    • kyber - 针对快速设备的调度

4. 分区处理

  • add_partition() - 添加分区
  • delete_partition() - 删除分区
  • check_partition() - 检查分区有效性

LevelDB复现之版本控制

发表于 2025-06-06 分类于 db

🧩 概要

🔍 struct FileMetaData字段详解：

🧱 字段详解：

字段名 类型 含义与作用
refs int 引用计数器：用于管理内存中的 FileMetaData 对象的生命周期，确保它不会被过早释放。
allowed_seeks int 允许的 seeks 次数（在触发 compaction 前）： 用于优化触发 compact 的时机。每当一个查询需要进行二分查找（seek）某个文件时，此计数会递减；当次数耗尽时会触发该文件的 compaction。 初始值为 2^30，表示无需紧急 compact。
number uint64_t SST 文件编号：唯一标识一个 SST 文件，对应磁盘文件名中的数字部分，如 000123.ldb。
file_size uint64_t SST 文件大小（以字节为单位）：用于统计和判断是否需要 compact（比如小文件可能被合并）。
smallest InternalKey SST 文件中最小的 InternalKey： 用于 key range 管理和快速查找（比如确定哪些文件与某个查询 key 有重叠）。
largest InternalKey SST 文件中最大的 InternalKey： 与 smallest 一起组成该文件的 key 范围。

在 LevelDB 中，Version 是一个非常核心的类，它表示 某一时刻整个数据库的快照，也就是整个 LSM 树的“视图”。每个 Version 都记录了当前数据库的所有 SST 文件在各个 Level 上的分布信息，支持读操作（Get）、迭代器构建（AddIterators）、重叠检测（GetOverlappingInputs）等功能。

我们可以把 Version 看作是数据库内部在某一时刻的 元数据状态快照，而 VersionSet 管理着这些快照的演进。

🧠 TableCache 是什么？

在 LevelDB 和 RocksDB 中，SST 文件是存储在磁盘上的。
为了避免频繁打开/关闭文件、重复解析数据结构，它们设计了一个轻量级的缓存系统，即 TableCache。

🧱 TableCache 的作用总结

功能 说明
📁 缓存 SST 文件的 Table 对象 减少打开文件（RandomAccessFile）的频率
🔢 按 file number 快速定位 所有 SST 文件都有唯一编号，TableCache 是 file_number → Table 的映射
🔍 加快读取速度 一次打开 SST 后，解析出的 index、filter block 会缓存在内存中
💥 节省内存使用 LRU 缓存机制，避免所有文件都常驻内存

🧬 TableCache 在读取流程中的位置

假设你执行一次 Get(key)，底层流程如下：

  1. MemTable 查找（可能命中）
  2. Immutable MemTable 查找（可能命中）
  3. SST 查找：
    • 根据 Version 里的 files_[] 决定要读哪些文件
    • 每个文件用 file_number 查找 TableCache
    • 如果命中：直接用已有的 Table* 访问
    • 如果没命中：
      • 打开文件 → RandomAccessFile
      • 解析 footer → block handle → index/filter block
      • 创建 Table 对象
      • 加入 TableCache

✅ 所以，TableCache 是 SST 文件到解析后的 Table 实例之间的缓存层，避免每次都从头加载。

📌 举例说明

假如你有 1000 个 SST 文件，查询一个 key foo：

  • 不使用 TableCache：每次都要打开文件、解析 footer、index block 等，非常慢。
  • 使用 TableCache：
    • 查询时命中缓存：O(1) 取出 Table*，直接查找
    • 不命中：只做一次解析，然后缓存起来，后续复用

🧼 补充一点：和 BlockCache 区别？

名称 缓存对象 默认大小 作用
TableCache SST 文件的 Table 实例 通常为打开文件数量 缓存文件和结构
BlockCache 数据块、filter block 可配置为几百 MB 缓存实际的 key-value 数据块

✅ 总结

TableCache 是 LevelDB 和 RocksDB 中的 SST 文件缓存层，负责将 file_number 映射到已经打开并解析好的 Table 实例，避免重复 I/O 和结构构建，是提升读取性能的关键组件。
