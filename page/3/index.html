SVR4 VFS/Vnode架构的发展历程

发表于 2025-05-13 分类于 文件系统

​ 在SVR3中引入的文件系统切换（FSS）架构、SunOS中的Sun VFS/vnode架构，以及两者在SVR4 中的融合，极大地改变了文件系统的访问方式和实现方式。在这一发展过程中，文件系统的类型数量显著增加，其中包括了像VxFS这样的商业文件系统的引入，这使得UNIX系统开始迈向企业级计算市场。

​ SVR4还引入了许多与文件系统相关的重要概念，例如将文件系统访问与内存映射文件相结合、引入目录名称查找缓存（DNLC），以及将传统的缓冲区缓存（buffer cache）与页面缓存（page cache）分离开来，这些也改变了I/O操作的执行方式。

本章将回顾促成SVR4实现的一系列发展历程。SVR4架构不仅成为Sun Solaris操作系统的基础，而且如今仍通过Caldera的 OpenUNIX项目以开源形式自由提供。

变革的必要性

如第 6 章所述，早期研究版的 UNIX 系统仅支持一种文件系统类型。内核与文件系统之间高度耦合的设计在当时运作良好，但随着需求的发展，这种模式已经逐渐无法满足新的扩展性与灵活性要求。

sentinel_for 是 C++20 标准中引入的一个概念（Concept），用于类型推断和范围算法中，检查一个类型是否可以作为容器的“哨兵”类型。

哨兵（Sentinel）的概念

在 C++ 中，容器通常由两部分构成：一个是迭代器，它指向容器中的元素，另一个是哨兵，它是一个标记，用来指示迭代是否结束。比如，在标准库中，std::vector 使用迭代器来遍历元素，而迭代器的末尾会指向 std::vector 的结束位置，即哨兵。

sentinel_for 详解

sentinel_for 是一个 概念（concept），用于检查某个类型是否适合作为一个容器的哨兵类型。在算法中，哨兵通常被用来和迭代器进行比较，以确定容器遍历是否完成。

sentinel_for 的定义

  • Sentinel 是一个容器的哨兵类型。
  • Iterator 是一个容器的迭代器类型。
  • sentinel_for 概念会检查是否可以将一个 Iterator 和一个 Sentinel 进行比较，并且比较结果能被转换为 bool。

解释

在容器中，通常会通过比较迭代器和哨兵来判断是否到达容器的末尾。例如，在 std::vector 中，end() 函数返回一个指向容器末尾的迭代器，它实际上是一个哨兵。当你在遍历容器时，通过与这个“哨兵”进行比较，来判断是否应该停止。

sentinel_for 概念主要用于检查某个类型是否可以作为容器的“结束标记”，即哨兵。它被广泛应用于算法中，以保证迭代器和容器的结束位置能够正确比较。

总结

  • sentinel_for 是一个 C++20 概念，用来检查某个类型是否能作为容器的哨兵类型。
  • 哨兵通常是用来与迭代器比较，表示容器的结束位置。
  • 这个概念用于标准库算法中，确保容器遍历的正确性和类型安全性。

SizedSentinelFor 是 C++20 中引入的一个概念（Concept），它扩展了 sentinel_for 概念，要求哨兵类型不仅可以与迭代器进行比较，还需要能够通过与迭代器的比较来确定容器的大小，或至少可以提供一个与容器大小相关的有效操作。

作用

SizedSentinelFor 主要用于支持那些可以根据容器大小直接计算或推断剩余元素个数的容器类型。例如，在一些容器中，哨兵不仅仅是表示结束的位置，它还可以携带关于容器大小的信息（例如，在数组或容器的迭代中可以通过 last - first 来计算元素的数量）。

定义

解释：

  • SizedSentinelFor 继承自 sentinel_for 概念，表示它至少满足哨兵和迭代器可以比较的要求。
  • 额外要求是，Iterator 和 Sentinel 之间可以进行减法操作，且减法的结果可以转换为 std::ptrdiff_t（通常是一个表示元素个数的整数类型）。

这意味着，除了能进行比较外，Iterator 和 Sentinel 还需要支持 i - s 操作，这样可以直接计算出迭代器与哨兵之间的距离（即容器的元素数量）。例如，在数组或者支持随机访问的容器中，容器的大小可以通过两个迭代器之间的距离来直接计算。

SizedSentinelFor 适用于需要进行元素计数操作的算法或数据结构。例如，某些标准库算法（如 std::distance）可以使用此概念来优化对容器大小的计算，而无需遍历整个容器。

标准I/O库

发表于 2025-05-13 分类于 文件系统

​ 操作系统提供基本文件访问系统调用（如 open、read 等），但用户在进行文件操作时，往往需要更高级的功能。标准 I/O库（stdio），作为ANSI C标准的一部分，提供额外的功能，避免重复造轮子。

​ 已有大量书籍就少标准I/O库提供的各类调用。而本章则采取不同的视角，通过剖析Linux标准I/O库的实现机制，介绍其核心结构、如何支持各类函数，以及库函数如何映射到底层UNIX系统调用。

应用程序是否使用标准I/O库，而不是直接调用底层系统调用，取决于应用的需求。如果需要更丰富的功能，且对性能要求不高，标准I/O库提供的丰富函数通常能够满足大多数程序员的需要。但如果性能是关键考量，或者需要更精细地控制I/O行为，那么理解文件系统的I/O实现，并绕过标准I/O库直接使用系统调用，会是更优的选择。

本章不会逐一讲解stdio提供的众多函数，而是聚焦于标准I/O库的底层实现机制。关于接口的详细介绍，可参考Richard Stevens所著的《UNIX高级环境编程》一书，或查阅Single UNIX Specification。

x86-64指令

发表于 2025-05-11 分类于 cpu

popcnt

​ popcnt指令是x86 CPU用来统计一个整数中二进制位为 1 的个数，全称是“population count”。

Opcode Instruction Op/En 64-Bit Mode Compat/Leg Mode Description
F3 0F B8 /r POPCNT r16, r/m16 RM Valid Valid POPCNT on r/m16
F3 0F B8 /r POPCNT r32, r/m32 RM Valid Valid POPCNT on r/m32
F3 REX.W 0F B8 /r POPCNT r64, r/m64 RM Valid N.E. POPCNT on r/m64
  1. Op/En (Operand Encoding)中的RM：
    • 表示该指令使用Reg-ModR/M编码格式，即指令操作数由寄存器（Register）和ModR/M字节（内存或寄存器操作数）组成。
    • 对于POPCNT指令，r16/r32/r64是目标寄存器，r/m16/r/m32/r/m64是源操作数。
  2. Compat/Leg Mode (Compatibility/Legacy Mode)：
    • 指指令在传统模式（Legacy Mode）或兼容模式（Compatibility Mode）下的支持情况。
      • Valid：表示该指令在该模式下可用。
      • N.E. (Not Enabled)：表示该指令在该模式下不可用（例如64位POPCNT在传统模式下不支持）。
    • 传统模式指非64位模式（如16位实模式或32位保护模式），而兼容模式是64位长模式下的32位子模式。
