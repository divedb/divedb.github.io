2.2 角色转变

​ 开始，所有节点都是Follower。每个节点会有一个随机的定时器。如果在这个时间之内没有收到Leader的消息，Follower角色变成Candidate。这个时候每个候选者就开始为自己拉票了，给所有其他的节点发送Request Vote请求。其他节点在收到Request Vote请求之后会基于一定的条件选择拒绝还是同意（看下文）。需要注意每个节点最多只能投票给其中一个人。每个Request Vote的发起者肯定厚颜无耻的投给了自己。假设有5个节点，每个candidate只需要拿到其他两个节点的票加上自己的一票就可以称为Leader了（$2+1 > 5/2$多数派原理）。如果在一个任期内没有选出Leader，会继续开始下一届选举（任期数加1）。需要注意，节点如果在上一个任期给其他节点投过票，在新的任期可以重新投票。

  1. 复制状态机

    • 相同的初始状态+相同的输入=相同的结束状态
    • 多个节点上，从相同的初始状态开始，执行相同的一串命令，产生相同的最终状态。
    • 在Raft中，leader将客户端请求（command）封装到一个个log entry中，将这些log entries复制到所有follow节点，然后大家按相同顺序应用log entries中的command，根据复制状态机的理论，大家的结束状态肯定是一致的。
    • 可以说，我们使用共识算法，就是为了实现复制状态机。一个分布式场景下的各节点间，就是通过共识算法来保证命令序列的一致，从而始终保持它们的状态一致，从而实现高可用的。（投票选主使一种特殊的命令）
    • 这里所谓拓展一点，复制状态机的功能可以更加强大。比如两个副本一个采用行存储的数据结构存储数据，另一个采用列存储，只要它们初始数据相同，并持续发给他们相同的命令，那么同一时刻从两个副本中读取到的结果也是一样的，这就是一种HTAP的实现方法（比如TiDB）。

  2. 状态简化

    • 在任何时刻，每一个服务器节点都处于leader，follower或candidate这三个状态之一。

    • 相比于Paxos，这一点就极大简化了算法的实现，因为Raft只需考虑状态的切换，而不用像Paxos那样考虑状态之间的共存和互相影响。

    • Raft把时间分割成任意长度的任期（term），任期用连续的整数标记。

    • 每一段任期从一次选举开始。在某些情况下，一次选举无法选出leader（比如两个节点收到了相同的票数），在这种情况下，这一任期会以没有leader结束；一个新的任期（包含一次新的选举）会很快重新开始。Raft保证在任意一个任期内，最多只有一个leader。

      • 选举新的领导者：在Raft协议中，领导者的选举是通过选举超时和投票来实现的。每个节点都有一个当前的任期号，节点会在选举超时后开始选举过程，递增当前的任期号，并发送请求投票消息给其他节点。通过更新任期号，可以触发新一轮的选举，选出新的领导者。
      • 防止过时的领导者产生冲突：如果一个过时的领导者（例如，任期号较小的领导者）试图在网络分区或故障恢复后继续操作，可能会与其他节点的领导者发生冲突，导致数据的不一致。通过更新任期号，可以防止过时的领导者继续操作，并确保新的领导者在网络重新连接后能够正确地处理请求。
      • 提供一致的视图：任期号用于标识一个特定的任期，所有节点都通过任期号来维护一致的视图。当一个节点接收到具有较新任期号的请求时，它会更新自己的任期号，以确保所有节点都具有相同的任期号，并且能够按照相同的规则进行操作。
      • 安全性和一致性保证：更新任期号有助于确保Raft协议的安全性和一致性。通过在每个任期中进行领导者选举，保证了领导者的合法性和合理性。同时，通过在任期号上达成一致，可以避免多个领导者同时存在，从而维持一致的操作和数据状态。

      总而言之，更新任期号在Raft协议中是为了维护领导者的稳定性和一致性，并提供安全性保证。通过更新任期号，可以触发选举过程、防止过时的领导者产生冲突、提供一致的视图，并确保协议的正常运行。

    • Raft算法中服务器节点之间使用RPC进行通信，并且Raft只有两种主要的RPC：

      • RequestVote RPC（请求投票）: 由candidate在选举期间发起。
      • AppendEntries RPC（追加条目）：由leader发起，用来复制日志和提供一条心跳机制。

    • 服务器之间通信的时候会交换当前任期号；如果一个服务器上的当前任期号比其他的小，该服务器会将自己的任期号更新为较大的那个值。

    • 如果一个candidate或者leader发现自己的任期号过期了，它会立即回到follower状态。

    • 如果一个节点接收到一个包含过期的任期号的请求，它会直接拒绝这个请求。

  3. 领导者选举

    • Raft内部有一种心跳机制，如果存在leader，那么它就会周期性的向所有follower发送心跳，来维持自己的地位。如果follower一段时间没有收到心跳，那么它就会认为系统中没有可用的leader，然后开始进行选举。
    • 开始一个选举过程后，follower先增加自己的当前任期号，并转换为candidate状态。然后投票给自己，并且并行的向集群中的其他服务器节点发送投票请求（RequestVote RPC）。

分布式共识算法(consensus algorithm)通常的做法就是在多个节点上复制状态机。分布在不同服务器上的状态机执行着相同的状态变化，即使其中几台机器挂掉，整个集群还能继续运作。

1. Raft leaders commit a no-op entry at the beginning of their term

no-op 日志即只有 index 和 term 信息，command 信息为空。也是要写到磁盘存储的。

具体流程是在 Leader 刚选举成功的时候，立即追加一条 no-op 日志，并立即复制到其它节点，no-op 日志一经提交，Leader 前面那些未提交的日志全部间接提交，问题就解决了。像上面的 kv 数据库，有了 no-op 日志之后，Leader 就能快速响应客户端查询了。

本质上，no-op 日志使 Leader 隐式地快速提交之前任期未提交的日志，确认当前 commitIndex，这样系统才会快速对外正常工作。

节点收到消息先判断Term还是Message Type

这是因为 Raft 算法的核心思想是基于任期（Term）的选举和领导者的角色确定。在任一给定的任期内，只有一个领导者可以被选举出来，其他节点则处于跟随者（Follower）状态。因此，处理消息时首先要确认消息的 Term 是否比当前节点的 Term 大。如果消息的 Term 比当前节点小，说明该消息可能是过期的，可以忽略。如果消息的 Term 比当前节点大，那么当前节点需要更新自己的 Term，并且切换到跟随者状态。

在确认 Term 合适后，才会考虑消息的 Type。Raft 消息的类型通常包括：

  1. Request Vote（选举投票请求）： 如果收到的消息是选举投票请求，那么节点会根据一定的规则（如候选者的 Term 大于等于自己的 Term）来判断是否投票给该候选者。
  2. Append Entries（日志追加请求）： 如果收到的消息是日志追加请求，通常需要对消息中的信息进行验证，如比较日志的索引和任期等。如果验证通过，跟随者会将该消息中的日志追加到自己的日志中。
  3. Response（响应消息）： 对于领导者发送的请求，收到的是对应的响应消息。领导者会根据响应来判断是否需要继续发送数据或者执行其他操作。

总之，首要的是通过消息的 Term 来判断是否需要更新节点状态，并在 Term 适当的情况下，根据消息的类型来执行相应的操作。这个处理流程保证了算法的正确性和一致性。

Back to Basics: const and constexpr

  • const

  • const_cast

    • Modifying a const declared object by removing its constness is undefine behavior

    • Don’t use a C-cast (int i = (int) myValue;), because it applies eventually a series of casts:

      static_cast => const_cast => reinterpret_cast

  • constexpr

    • Constant expressions

      • can be evaluated at compile time
      • give the compiler deep insight
      • are implicit thread-safe

    • Variables

      constexpr double myDouble = 5.2;

      const int myInt = 5;

      • are implicit const.
      • are implicit thread-safe => A data race requires shared mutable state.
      • const variables are implicit constexpr when initialized with a constant expression.

    • Functions

      • must resolve each dependency at compile time.
      • can have variables that must be initialized by constant expressions.
      • cannot have static and thread_local variables.
      • have the potential to run at compile time. => Must tun at compile time when used in a constant expression.
      • are pure.

  • consteval

    generates an immediate function.

  • constinit

    constinit guarantess that a variable with static storage duration is initialized at compile time. This variable is still mutable.

    • Global objects, or objects declared with static or extern, have static storage duration.
    • Objects with a static storage duration are allocated at the program start and deallocated at its end.

Pragma系列

使用#pragma GCC system_header可以告诉编译器忽略对这些头文件的警告和错误，这有助于减少编译器输出的噪音，提高编译速度，并且可以使编译器更容易地优化代码。但是需要注意的是，使用这个指令时，如果你在这些头文件中有自己的错误，则编译器将无法发现并报告这些错误。

  • -Wall 选项表示开启所有警告信息，包括常见的错误和潜在的问题。但它并不会将所有警告都视为错误，所以编译器会继续编译代码。

  • -Wextra 选项与 -Wall 类似，它会开启更多的警告信息，包括一些非常规的警告。例如，它会检测不必要的分号、函数参数的类型不匹配等。同样地，-Wextra 也不会将所有警告都视为错误。

  • -Werror 选项将所有警告都视为错误，这意味着编译器会在发现任何警告时停止编译，并将其视为编译错误。这可以强制开发人员修复代码中的所有警告，确保代码的质量和稳定性。

  • -Wshadow 是 GCC 和 Clang 编译器的一个编译选项，用于检测变量名遮蔽（shadowing）的问题。

因此，-Wall 和 -Wextra 是开启警告信息的选项，而 -Werror 则是将警告转化为错误的选项。使用 -Werror 可以确保代码的质量，但需要开发人员花费更多的时间来修复所有的警告。

什么时候不生成default constructor

在以下情况下，编译器不会自动生成默认构造函数：

  1. 如果类中已经显式声明了任何构造函数（包括拷贝构造函数和移动构造函数），则编译器不会生成默认构造函数。

    error 1

    error 2

  2. 如果类的成员变量有一个没有默认构造函数的非静态成员变量，则编译器不会生成默认构造函数。这是因为编译器需要调用每个成员变量的构造函数来初始化对象，而没有默认构造函数的成员变量无法自动初始化。

  3. 如果类继承了一个没有默认构造函数的基类，并且没有在构造函数的初始化列表中显式调用基类的构造函数，则编译器不会生成默认构造函数。这是因为在创建派生类对象时，需要先创建基类对象，而如果基类没有默认构造函数，则无法创建基类对象，也就无法创建派生类对象。

  4. 如果类中存在虚函数，且没有默认实现，则编译器不会生成默认构造函数。这是因为如果类中存在虚函数，编译器会自动生成一个虚函数表，并将指向虚函数表的指针存储在对象中。而如果没有默认实现，那么生成的虚函数表中就会有一个空指针，这可能导致程序运行时错误。

6. IMUL Instruction

  • The IMUL (signed multiply) instruction performs signed integer multiplication.
  • It has the same syntax and uses the same operands as the MUL instruction.
  • What is different is that it preserves the sign of the product.
  • IMUL sets the Carry and Overflow flags if the high-order product is not just a sign extension of the low-order product.
  • The Overflow flag OF is normally used for interpreting signed arithmetic.
