    Ext2文件系统

    发表于 2024-10-22 更新于 2025-04-30 分类于 linux

    目录：

    • Blocks
    • Block Groups
    • Directories
    • Inodes
    • Superblocks
    • Symbolic Links

    ​ Ext2文件系统使用块（block）作为基本的存储单位，使用inode（索引节点）来追踪文件和系统对象。它将磁盘逻辑上划分为更易管理的块组（block group），使用目录（directory）来提供文件的层级组织结构，并通过块位图和inode位图来追踪哪些块和inode已被分配。此外，它还使用超级块（superblock）来定义文件系统的参数和整体状态。

    Ext2与传统Unix文件系统有很多相似的特性。它的规范中预留了

    • 对访问控制列表（ACLs）
    • 文件碎片（fragments）
    • 反删除（undeletion）
    • 压缩（compression）

    等功能的支持空间。同时，它还设计了一套版本机制，以尽可能兼容的方式引入新特性（比如日志功能），这也为后续如 Ext3和Ext4的发展打下了基础。

    块（Blocks）

    ​ 被格式化为Ext2文件系统的分区、磁盘、文件或块设备会被划分为“块（block）”的扇区组。这些块进一步组合成更大的单元，称为块组（block groups）。

    块的大小通常在格式化磁盘时决定，这将影响性能、最大文件大小以及最大文件系统容量。常见的块大小包括1KB、2KB、4KB和8KB，不过在超级块（superblock）中预留了字段（如s_log_block_size），支持的块最大可达到 **1024 × (2³¹ - 1)**。

    根据具体实现的不同，一些体系结构可能会对支持的块大小加以限制。例如，在DEC Alpha架构上运行的Linux 2.6 实现可以使用8KB的块，而相同实现若运行在Intel 386处理器上，则最大支持的块大小为4KB。

    • file system blocks：因为Ext文件系统使用32位整数存储块编号，因此最大编号是INT_MAX（2,147,483,647）。

    块组（Block Groups）

    这一定义来自于Linux内核文档。

    为了减少碎片并最小化在读取大量连续数据时磁头的寻道次数，块（Blocks）被聚集成块组（Block Groups）。每个块组的信息保存在描述符表（Descriptor Table）中，该表位于超级块（Superblock）之后的一个或多个块中。

    在每个块组的起始处，保留了两个块用来存储：

    • 块使用位图（Block Usage Bitmap）
    • inode使用位图（Inode Usage Bitmap）

    它们分别表示哪些块和哪些inode正在被使用。由于每个位图仅限于一个块的大小，因此一个块组的最大大小是块大小的8倍。在每个块组中，位图之后的块被指定为该组的inode表（Inode Table），其余的则是数据块（Data Blocks）。

    当分配数据块时，系统会尝试将它们分配在与包含它们的inode属于同一个块组中，以此提高访问效率。

    目录（Directories）

    本定义来自Linux内核文档，略有修改。

    目录是文件系统中的一种对象，和普通文件一样拥有一个inode。目录文件包含若干记录，每条记录将一个名称与一个 inode编号关联起来。文件系统的后续版本还会在记录中编码对象的类型（如文件、目录、符号链接、设备、FIFO、套接字等），以避免必须通过inode本身来判断类型。

    在分配inode时，系统应尽量将新文件的inode分配在与其所在目录相同的块组中，以提高性能。

    最早的Ext2版本使用单向链表（singly-linked list）来存储目录下的文件名；而较新的版本则可以使用哈希表和二叉树来提高查找效率。

    另外还需要注意：当目录变大时，会分配额外的块来存储新增的文件记录；但当文件名被删除后，有些实现并不会回收这些额外的块。

    Inodes

    本定义来自Linux内核文档，略有修改。

    inode（索引节点）是Ext2文件系统中的一个基本概念。文件系统中的每个对象都由一个inode表示。inode结构中包含指向保存该对象数据的文件系统块的指针，以及该对象所有除名称之外的元数据。对象的元数据包括权限、所有者、所属组、标志、大小、所占用的块数、访问时间、更改时间、修改时间、删除时间、链接数、碎片信息、版本号（用于NFS）以及扩展属性（EAs）和/或访问控制列表（ACLs）。

    inode结构中有一些预留字段，目前尚未使用，还有一些字段被重载使用。其中有一个字段在inode为目录时用于存储目录 ACL，而当inode为常规文件时则用于存储文件大小的高32位（从而允许文件大小超过2GB）。另有一个translator字段在 Linux下未被使用，但在HURD中用于引用将用于解释该对象的程序的inode。大多数其他预留字段已被Linux和HURD用于扩展的所有者和组字段，HURD还需要一个更大的mode字段，因此利用剩余的字段存储额外的位。

    在inode中，有指向前12个块的指针，用以存放文件的数据。接着，还有一个指向间接块（该块包含下一组块的指针）的指针，一个指向双重间接块（存储间接块指针）的指针，以及一个指向三重间接块（存储双重间接块指针）的指针。

    此外，还存储有一些文件系统特定的行为标志，这些标志允许对每个文件实现特定的文件系统行为。这些标志包括安全删除、不可删除、压缩、同步更新、不可变性、只追加、可转储、无访问时间更新、索引目录以及数据日志记录等。

    许多文件系统特定的行为标志（例如日志功能）已在更新的文件系统如Ext3和Ext4中实现，而一些其他标志仍在开发中。

    所有inode都存储在inode表中，每个块组拥有一个inode表。

    超级块（Superblocks）

    本定义来源于Linux内核文档，并作了一些小的改动。

    超级块包含了文件系统配置的所有信息。超级块中的内容包括文件系统中inode和块的总数及空闲数量、每个块组中的inode 和块数量、文件系统的挂载时间（以及是否被干净卸载）、最后修改时间、文件系统版本及创建该文件系统的操作系统等信息。

    超级块的副本位于设备起始处偏移1024 字节的位置，这是挂载文件系统时必不可少的部分。由于超级块的重要性，文件系统会在各个块组中保存它的备份副本。

    在Ext2的第一个版本（修订版 0）中，每个块组的起始位置都会保存一份超级块副本，以及组描述符块的备份。但在大型文件系统中，这种做法会占用大量空间，因此后续的修订版本提供了“稀疏超级块（sparse superblock）”功能，仅在某些特定的块组中保存超级块的副本。被选中的块组通常是编号为0、1，以及 3、5、7 的幂次方的那些。

    文件系统的修订版1及更高版本还引入了额外的字段，例如卷名、唯一识别号、inode大小，以及用于可选文件系统功能的空间（用于存储配置数据）。

    超级块中的所有字段（就像Ext2中所有其他结构一样）都以小端格式（little endian）存储在磁盘上，因此文件系统可以在不同的机器之间移植，而无需关心最初是在哪种机器上创建的。

    符号链接（Symbolic Links）

    符号链接（也称为symlink或软链接）是一种特殊类型的文件，它包含一个指向另一个文件或目录的引用路径，可以是绝对路径或相对路径，并参与路径解析的过程。

    对于大多数操作而言，符号链接是透明的：程序在读取或写入一个由符号链接指定的文件时，会像直接操作目标文件那样工作。然而，一些需要特别处理符号链接的程序（如备份工具）则可能识别并直接操作它们。

    符号链接本身只包含一个文本字符串，该字符串会被操作系统解释为对其他文件或目录的路径引用。它本身是一个独立的文件，即使目标不存在也可以单独存在。符号链接不会影响inode的引用计数：

    • 如果删除一个符号链接，其指向的目标不会受到影响；
    • 如果目标被移动、重命名或删除，符号链接仍然存在，但会变成指向不存在对象的链接，这种链接通常被称为“孤立链接”或“悬空链接”（dangling link）。

    符号链接本身也是文件系统对象，也有inode。对于小于60字节的符号链接，其链接路径数据会直接存储在inode的字段中（这些字段原本用于保存数据块指针）。这是一个非常有效的优化，因为它避免了为短链接额外分配磁盘块，而大多数符号链接本来就短于 60 个字符。

    此外，符号链接还可以跨分区或跨文件系统，指向其他位置的文件或目录。

    为什么是“60 字节”？

    这个数字取决于：

    • inode中“数据块指针字段”的总字节数（通常为15个指针 × 4字节 = 60字节），
    • 所以，只要路径字符串小于这个长度，就可以塞进inode里。

    Chapter 3. Disk Organization

    ​ 一个Ext2文件系统从偏移1024字节处的超级块（superblock）开始。对于使用1KB块大小格式化的卷来说，这是第1块；而对于更大的块大小，该偏移位于第0块的内部。需要注意的是，无论块大小如何，超级块的大小始终是固定的。

    在超级块之后的块中，是块组描述符表（Block Group Descriptor Table），它提供了关于卷如何划分为块组的信息，并指明了每个块组中的 inode 位图、块位图以及 inode 表的位置。

    在 Ext2 的修订版 0（revision 0）中，每个块组包含：

    • 一份超级块副本
    • 一份块组描述符表副本
    • 一个块位图
    • 一个inode位图
    • 一个inode表
    • 以及数据块（data blocks）。

    从修订版 1起，引入了稀疏超级块（sparse superblock）功能，仅在特定块组中保存超级块和块组描述符表的副本。
    所有块组仍然包含块位图、inode位图、inode表和数据块。超级块的副本通常保存在块组0、1以及3、5、7的幂次块组中。

    每个块组中的块位图和inode位图各被限制为1个块大小，这对每个块组内可包含的总块数做出了限制。（可参考“块大小影响表”以了解更多信息。）

    每个数据块还可以被进一步划分为“碎片（fragments）”。但截至Linux内核版本 2.6.28，碎片机制仍未被实现，因此建议将碎片大小设置为等于块大小，以保持系统兼容性。

    思考：前1024字节的作用？

    ✅ 在Ext2中，第一个超级块（superblock）总是从偏移量 1024 字节开始，也就是说：

    • 前1024 字节是留空的，通常不会由文件系统使用
    • 它们被称为Boot Sector / Boot Record
    • 如果这个分区是启动分区，BIOS/UEFI会在这段区域加载引导代码

    📦 那么，这前1024字节里都有什么？

    区域范围（字节） 内容 说明
    0 ~ 510 可选的MBR/引导代码 如果这是启动分区，可存放bootloader
    511 ~ 512 引导扇区签名（0x55AA） 标志引导扇区有效
    512 ~ 1023 可选的其他引导信息 某些系统会放置二级引导程序

    ⚠️ 如果这个分区不打算启动，那么这1024字节可以是全0，文件系统不会使用这段区域。

    🧠为什么Ext2把superblock放在offset 1024？

    这是个历史遗产，目的是兼容引导加载器。历史上DOS/BIOS 兼容分区会用前几个扇区做启动代码，Ext系列设计时为了兼容，也把文件系统真正的数据往后挪了一点。

    另外，这种设计也便于跨平台兼容，比如：

    • 多操作系统共存
    • GRUB/LILO引导程序可以放心写前1024字节
    • 文件系统在偏移1024开始，不会被破坏

    block 0总是指向磁盘或分区的第一个扇区，如果存在引导记录（boot record），它就会存储在这个位置。

    超级块（superblock）始终位于磁盘或分区起始偏移量的1024字节处。

    • 在以1KB 块大小格式化的文件系统中，这个位置是block 1；
    • 而在更大的块大小中，它仍然在block 0的1024字节偏移处。

    磁盘上的布局是可预测的，前提是了解几个关键信息：

    • block size（块大小）
    • blocks per group（每个块组的块数）
    • inodes per group（每组的 inode 数量）

    这些信息都存储在超级块中，或可以由超级块结构推导出来。

    不过，除非镜像是以可控参数精心构建的，否则不要假定磁盘上各种结构的位置（除了超级块）。应该始终优先加载超级块。

    注意：在使用 1KiB block size 的文件系统中，block 0不属于块组 0。原因是块组 0 必须从包含超级块的块开始。因此：

    • 在 1KiB block 系统中，block group 0 从 block 1 开始；
    • 在更大 block size（比如 2KiB、4KiB）系统中，block group 0 从 block 0 开始（因为 1024 字节落在 block 0 中）。

    有关详细信息，可以参阅超级块字段 s_first_data_block。

    Superblock（超级块）

    Superblock总是位于格式化为Ext2及其后续变种（Ext3、Ext4）的文件、块设备或分区的起始偏移1024字节处。

    其结构从Ext2到Ext3和Ext4基本保持一致，仅做了些小的调整。

    • s_inodes_count：一个32位的值，表示文件系统中inode（包括已使用和未使用的）的总数。此值必须小于或等于 (s_inodes_per_group * 块组数量)。 它必须等于每个块组中定义的inode数的总和。

    • s_blocks_count：一个32位的值，表示文件系统中块（包括已使用、未使用和保留块）的总数。此值必须小于或等于 (s_blocks_per_group * 块组数量)。如果最后一个块组由于卷大小的原因块数少于s_blocks_per_group，那么这个值可能小于上述计算结果。 它必须等于每个块组中定义的块数的总和。

    • s_r_blocks_count：一个32位的值，表示为超级用户保留的块总数。当某个用户（无论是恶意还是非恶意）将文件系统填满时，这个值最为有用；超级用户仍可以使用这部分预留的块，用于编辑和保存配置文件。

    • s_free_blocks_count：一个32位的值，表示空闲块的总数，包括保留块的数量（见 s_r_blocks_count）。它是所有块组中空闲块数量的总和。

    • s_free_inodes_count：一个32位的值，表示空闲inode的总数。它是所有块组中空闲inode数的总和。

    • s_first_data_block：一个32位的值，表示第一个数据块的编号，也就是说包含超级块结构的块的编号。

      注意：

      • 对于块大小大于1KB的文件系统，此值始终为0。

      • 对于块大小为1KB的文件系统，此值始终为1。

    ​ 超级块始终从磁盘的第1024字节开始，这通常是第三个扇区的第一个字节。

    • s_log_block_size：这个32位值表示需要将1024向左移动的位数，以计算块大小。此值只能为非负数。

    ​ 常见的块大小包括1KB、2KB、4KB和8KB。

    ​ 注意：
    ​ 在 Linux（至少到 2.6.28 版本）中，块大小必须不小于块设备的扇区大小，且不能大于体系结构支持的内存页大小。

    • s_log_frag_size：这个32位值用于计算碎片大小（fragment size），其方法是将1024向左或右移动相应位数

    ​ 注意：

    ​ 截至Linux 2.6.28，Ext2分区不支持碎片大小小于块大小的情况，因为此特性似乎并未被实现。

    • s_blocks_per_group：一个32位整数，表示每个块组（block group）中包含的块（block）数量。结合 s_first_data_block 字段，可以用来计算各个块组的边界。由于卷大小限制，最后一个块组的实际块数可能少于该字段中指定的数量。

    • s_frags_per_group：一个32位整数，表示每个块组中包含的片段（fragment）数量。这个值也用于确定每个块组中块位图（block bitmap）的大小。

    • s_inodes_per_group：一个32位整数，表示每个块组中包含的索引节点（inode）数量。同样也用于计算每个块组中 inode位图（inode bitmap）的大小。

      注意：每个块组的inode数量不能超过（块大小 × 8），因为inode位图必须能完全存放在一个块内。此外，该字段的值必须是可以整除单个块中可容纳的inode数量的，即必须是 ((1024 << s_log_block_size) / s_inode_size) 的整数倍。

    • s_mtime：文件系统最后一次被挂载（mount）的时间，使用POSIX定义的Unix时间格式表示。

    • s_wtime：文件系统最后一次被写入（write）的时间，同样采用POSIX的Unix时间格式。

    • s_mnt_count：一个16位整数，记录自上次完整校验以来，文件系统被挂载的次数。

    • s_max_mnt_count：一个16位整数，表示在进行一次完整校验前，文件系统允许挂载的最大次数。

    • s_magic：一个16位整数，用于标识文件系统类型是否为Ext2。这个值固定为EXT2_SUPER_MAGIC（0xEF53）。

    • s_state：

      一个16位整数，表示文件系统当前的状态：

      • 当文件系统被挂载时，状态会被设置为EXT2_ERROR_FS。

      • 当文件系统被正常卸载后，状态会被设置为EXT2_VALID_FS。

      如果在挂载过程中发现状态是EXT2_ERROR_FS，这表示文件系统上次未能正常卸载，可能存在错误，需要修复。通常在 Linux系统中，这意味着需要运行fsck工具进行检查。

    s_state 定义表：

    常量名 值 说明
    EXT2_VALID_FS 1 正常卸载
    EXT2_ERROR_FS 2 检测到错误

    • s_errors：一个16位整数，指定在检测到错误时，文件系统驱动应采取的应对措施。

      s_errors 定义表：

    • s_minor_rev_level：一个16位整数，表示当前主版本（revision level）下的小版本号。

    • s_lastcheck：上一次进行文件系统完整性检查（fsck）的时间，采用POSIX定义的Unix时间格式。

    • s_checkinterval：文件系统允许两次完整性检查之间的最大时间间隔（Unix时间），超过该间隔后应强制进行一次检查。

    • s_creator_os：一个32位整数，表示创建该文件系统时使用的操作系统。定义的取值包括（未列出完整取值，通常有 Linux、GNU Hurd、FreeBSD等）。

    • s_def_resuid：一个16位整数，表示预留块的默认用户ID。

    块组描述符表（Block Group Descriptor Table）

    ​ 块组描述符表是一个由多个块组描述符组成的数组，用于定义所有块组的相关参数。它提供了关于inode位图、inode表、块位图的位置，以及空闲块数、空闲inode数和其他有用信息。

    该表紧随超级块之后，即为第一个块。对于使用1KB块大小的文件系统，这是第3个块；而对于使用2KB或更大块大小的文件系统，则是第2个块。每一个超级块的备份副本旁边也保存了块组描述符表的备份。

    根据块组的数量，该表可能会占用多个块。在存疑时，应以超级块中的信息为准。

    块组描述符的结构如下：

    表 3.12 块组描述符结构（Block Group Descriptor Structure）

    偏移量（字节） 大小（字节） 描述
    0 4 bg_block_bitmap
    4 4 bg_inode_bitmap
    8 4 bg_inode_table
    12 2 bg_free_blocks_count
    14 2 bg_free_inodes_count
    16 2 bg_used_dirs_count
    18 2 bg_pad
    20 12 bg_reserved

    文件系统中的每个块组都有一个对应的group_desc结构体。每个结构体只描述其对应的那个块组，与其他块组无关。块组描述符表汇总了所有块组的相关信息。

    注意： 所有“块 ID”都是绝对块编号。

    字段说明：

    • bg_block_bitmap

      表示该块组中“块位图”所在的第一个块的32位块ID。实际的位图数据从该块开始向后分布在连续的块中。

    • bg_inode_bitmap

      表示该块组中“inode 位图”所在的第一个块的32位块ID。

    • bg_inode_table

      表示该块组中“inode 表”所在的第一个块的32位块ID。

    • bg_free_blocks_count

      该块组中空闲块的总数（16位）。

    • bg_free_inodes_count
      该块组中空闲inode的总数（16位）。

    • bg_used_dirs_count

      表示该块组中已分配给目录的inode数量（16位）。

    • bg_pad

      用于结构体的对齐（16位填充字段）。

    • bg_reserved

      为未来扩展保留的 12 字节空间。

    块位图（Block Bitmap）

    ​ 在小型文件系统中，每个块组的“块位图”通常位于该组的第一个块；如果存在超级块备份，则为第二个块。其精确位置可通过该组描述符中的bg_block_bitmap字段获得。

    每一位表示该块组中一个块的使用状态：

    • 1表示“已使用”，
    • 0表示“空闲”。

    例如：

    • 第1个块由第0字节的第0位表示；
    • 第2个块由第0字节的第1位表示；
    • 第8个块对应的是第0字节的最高位（第7位）；
    • 第9个块则对应第1字节的最低位（第0位）。

    Inode 位图（Inode Bitmap）

    ​ “Inode 位图”的工作方式与“块位图”相似，不过每一位表示的是inode的使用状态，而不是块。由于inode编号从 1 开始，因此第一个块组的inode位图中的第一位代表的是inode编号1。

    每个块组有一个inode位图，其位置可通过相应描述符中的bg_inode_bitmap字段确定。

    在inode表创建时，所有保留的inode会被标记为“已使用”。在版本0中，前11个inode是保留inode。

    Inode 表（Inode Table）

    ​ Inode表用于记录系统中每一个目录、普通文件、符号链接或特殊文件的信息，包括它们的位置、大小、类型和访问权限等。inode本身并不包含文件名，文件名存储在目录文件中。

    每个块组有一个inode表，其起始块号由相应的bg_inode_table字段指定。每个表中有s_inodes_per_group个inode。

    每个inode表示系统中的一个“文件”，这个“文件”可以是目录、套接字、缓冲区、字符设备、块设备、符号链接或普通文件。也就是说，inode是描述一个实体的结构体，记录其在磁盘中的位置、大小、权限和所有者等信息。

    • i_ctime：32位值，表示自1970年1月1日以来该inode创建的时间（秒）。

    • i_mtime：32位值，表示自1970年1月1日以来最后一次修改该inode的时间（秒）。

    • i_dtime：32位值，表示自1970年1月1日以来该inode删除的时间（秒）。

    • i_links_count：16位值，指示该inode被链接的次数（被引用的次数）。大多数文件的链接计数为1，具有硬链接的文件每个硬链接都会增加计数。符号链接不会影响inode的链接计数。当链接计数为0时，inode及其所有关联的块会被释放。

    • i_blocks：32位值，表示为包含该inode数据而保留的512字节块的总数，无论这些块是否被实际使用。这些保留块的块号存储在i_block数组中。由于该值表示的是512字节块，而不是文件系统块，因此该值不应直接用作i_block数组的索引。相反，i_block数组的最大索引应通过i_blocks / ((1024<<s_log_block_size)/512)计算，简化后为 i_blocks/(2<<s_log_block_size)。

    • i_flags：32位值，指示在访问该inode数据时，Ext2实现应该如何行为。

    EXT2_UNRM_FL 0x00000002 记录用于恢复删除的文件
    EXT2_COMPR_FL 0x00000004 压缩文件
    EXT2_SYNC_FL 0x00000008 同步更新
    EXT2_IMMUTABLE_FL 0x00000010 不可更改的文件
    EXT2_APPEND_FL 0x00000020 仅追加操作
    EXT2_NODUMP_FL 0x00000040 不进行转储/删除文件
    EXT2_NOATIME_FL 0x00000080 不更新 .i_atime
    – 压缩使用保留 –
    EXT2_DIRTY_FL 0x00000100 脏文件（已修改）
    EXT2_COMPRBLK_FL 0x00000200 压缩块
    EXT2_NOCOMPR_FL 0x00000400 访问原始压缩数据
    EXT2_ECOMPR_FL 0x00000800 压缩错误
    – 压缩标志结束 –
    EXT2_BTREE_FL 0x00001000 B树格式目录
    EXT2_INDEX_FL 0x00001000 哈希索引目录
    EXT2_IMAGIC_FL 0x00002000 AFS 目录
    EXT3_JOURNAL_DATA_FL 0x00004000 日志文件数据
    EXT2_RESERVED_FL 0x80000000 为 ext2 库保留

    • i_osd1：32位操作系统相关值。

    • Hurd：32位值，标记为“translator”。

    • Linux：当前保留的32位值。

    • Masix：当前保留的32位值。

    • i_block：15个32位块编号，指向包含此 inode 数据的块。前12个块为直接块。数组中的第13个条目是第一个间接块的块号；该块包含一个包含数据块 ID 的块数组。因此，文件的第13个块将是间接块中包含的第一个块 ID。在 1KiB 块大小下，文件数据的块 13 到 268 都包含在该间接块中。第14个条目是第一个双重间接块的块号；该块包含一个间接块 ID 数组，每个间接块包含一个包含数据的块数组。在 1KiB 块大小下，每个双重间接块将包含 256 个间接块，每个间接块包含 256 个直接块，总共 65536 个数据块。第15个条目是三重间接块的块号；该块包含一个双重间接块 ID 数组，每个双重间接块包含一个间接块数组，每个间接块包含一个直接块数组。在 1KiB 文件系统中，每个三重间接块总共包含 16777216 个块。在 Ext2 的原始实现中，数组中的值为 0 时表示终止，并且没有更多的块被定义。在稀疏文件中，某些块已经分配，而其他块尚未分配，值 0 用于指示哪些块尚未分配。

    • i_generation：32位值，用于表示文件版本（NFS使用）。

    • i_file_acl：32位值，指示包含扩展属性的块号。在修订版0中，此值始终为0。

      注：有关Linux上ACL补丁和实现状态的更多信息，请访问 http://acl.bestbits.at/

    • i_dir_acl：在修订版0中，此32位值始终为0。在修订版1中，对于常规文件，此32位值包含64位文件大小的高32位。

      注：如果文件不是常规文件（例如块设备、目录等），Linux将此值设置为0。理论上，此值可以指向包含目录或特殊文件扩展属性的块。

    • i_faddr：32位值，指示文件碎片的位置。

      注：在Linux和GNU HURD中，由于不支持碎片，因此此值始终为0。在Ext4中，此值现在标记为过时。理论上，这应包含托管实际碎片的块号。碎片号及其大小将包含在i_osd2结构中。

    Inode i_osd2结构体

    该结构体为操作系统相关的96位结构体。

    Hurd系统

    表 3.17. Inode i_osd2 结构体（Hurd）

    偏移量（字节） 大小（字节） 描述
    0 1 h_i_frag
    1 1 h_i_fsize
    2 2 h_i_mode_high
    4 2 h_i_uid_high
    6 2 h_i_gid_high
    8 4 h_i_author
    • h_i_frag：8位片段编号。由于GNU HURD不支持文件片段，此值始终为0。在Ext4中已废弃。
    • h_i_fsize：8位片段大小。与上类似，始终为0，已在Ext4中废弃。
    • h_i_mode_high：文件权限mode字段的高16位。
    • h_i_uid_high：用户 ID 的高16位。
    • h_i_gid_high：组ID的高16位。
    • h_i_author：分配给该文件的作者用户ID，32位。如果该值为-1，则使用标准的POSIX用户ID。

    Linux 系统

    表 3.18. Inode i_osd2 结构体（Linux）

    偏移量（字节） 大小（字节） 描述
    0 1 l_i_frag
    1 1 l_i_fsize
    2 2 保留字段
    4 2 l_i_uid_high
    6 2 l_i_gid_high
    8 4 保留字段

    • l_i_frag：8位片段编号。Linux不支持片段，因此该值总为 0。

      注意： 如果i_faddr的值为 0，新的Ext2实现应完全忽略该字段。

      重要提示：在Ext4中，此字段与l_i_fsize一起表示inode数据块数的高16位（共48位）。

    • l_i_fsize：8位片段大小。含义同上。

      注意和提示： 与l_i_frag说明相同。

    • l_i_uid_high：用户ID的高16位。

    • l_i_gid_high：组ID的高16位。

    Masix 系统

    表 3.19. Inode i_osd2 结构体（Masix）
